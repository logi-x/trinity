## The two highest-risk surfaces for this feature class — both verified CLOSED

**1. Tar import path traversal / arbitrary file write — SAFE.**
`restore_from_tar` (`snapshot.py:77`) is triple-guarded per member: reject absolute paths, reject `..` in `Path(name).parts`, require allowlist match (`data/**`), then a post-`resolve()` `target.relative_to(home_resolved)` escape check (catches traversal through *pre-existing* symlinks in `data/`). It never creates symlinks — only `write_bytes` on regular-file members (`extractfile` returns `None` for symlinks → skipped). The backend import endpoint hard-codes `paths=["data/**"]`, so the agent can't widen its own allowlist.

**2. Command injection via `materialize_data_paths` heredoc — SAFE (no privilege boundary crossed).**
`materialize_trinity_yaml_list` builds `bash -c "<heredoc>"` embedding template-supplied `data_paths`. Three independent reasons it doesn't yield injection:
- docker-py 7.1.0 runs string commands through `shlex.split()` (confirmed in `exec_create` → `utils.split_command`), **not** an intermediate `/bin/sh` — so no `$()`/backtick/`;` expansion at the transport layer.
- The heredoc delimiter is single-quoted (`<<'DATAPATHS_EOF'`) → bash performs no expansion of the body; `yaml.safe_dump` escapes newlines, so a value can't smuggle a bare delimiter line.
- It runs via `execute_command_in_container("agent-{name}", …, user="developer")` — inside the agent's **own** container, which the template already fully controls. No cross-tenant / backend boundary is crossed.

The `.gitignore` append path (`_build_gitignore_append_command`) `shlex.quote`s every pattern and the whole script. MCP/URL paths `encodeURIComponent` the agent name. Auth: both endpoints use `OwnedAgentByName` (404 if unregistered, 403 unless owner/admin); `agent_name` must resolve to a real registered agent before any container op.

### MEDIUM
**M1 — Import buffers the entire upload into memory before the size check.** — ✅ **RESOLVED 2026-06-21** (see Resolution below).
`import_agent_data` (`agent_data.py`) does `blob = await tarball.read()` and only then `if len(blob) > AGENT_DATA_EXPORT_MAX_BYTES: 413`. Starlette spools the multipart body to a temp file, but `.read()` pulls the whole thing into a `bytes` in RAM (and it's buffered again to ship to the agent-server). An authenticated owner/admin can POST a multi-GB body and OOM the backend — platform-wide impact (affects all tenants, not just the attacker's agent), which is why this rates above "self-harm". Export, by contrast, does this correctly: `_drain_stream_to_file` enforces the cap *during* the write loop.
*Mitigations already present:* owner/admin gate + per-agent Redis op-lock. *Recommended fix:* reject early on `Content-Length` over cap, or stream the upload to a temp file with the cap enforced per-chunk (mirror the export path).

### LOW
**L1 — `data_paths` values are not charset-validated before the heredoc write.** — ✅ **RESOLVED 2026-06-21** (see Resolution below).
A `"` or control character in a template's `data_paths` entry breaks the `bash -c "…"` shlex tokenization and fails `materialize_data_paths`. It is **not** injection (see surface #2), and the call is wrapped in try/except → logged, non-fatal, agent creation continues. Robustness nit: validate `data_paths` entries against a safe glob charset at template load so a malformed template fails loudly rather than silently skipping its ignore rules.

### INFORMATIONAL
**I1 — Data export/import MCP tools are gated only by backend ownership, not MCP-layer `agent_permissions`.**
Like the existing lifecycle/credential MCP tools (`start_agent`, `delete_agent`, `export_credentials`, …) — and unlike `chat_with_agent`/`list_agents` — these tools rely on backend `OwnedAgentByName` rather than an MCP-layer `agent_permissions` check. An agent-scoped key can therefore export/import a *sibling* agent owned by the same user without an explicit grant. This is consistent with the established pattern for owner-operation tools and is **not** a new gap; noted only so a future "require `agent_permissions` on data movement" decision is deliberate.

## Resolution (2026-06-21)

**M1 — fixed.** `import_agent_data` (`routers/agent_data.py`) no longer does `blob = await tarball.read()`. The upload is now:
- fast-rejected up front when the declared `Content-Length` exceeds `AGENT_DATA_EXPORT_MAX_BYTES` (413);
- streamed to a capped temp file under `_TMP_DIR` via the new async helper `_drain_upload_to_file`, which raises `_ExportTooLarge` the moment the running total crosses the cap (mirrors the export-side `_drain_stream_to_file`) — so the body is never materialized as one large in-memory `bytes`;
- forwarded to the agent restore primitive from an open file handle (`files={"tarball": ("data.tar", fh, …)}`) so httpx streams it rather than re-buffering;
- cleaned up in a `finally` (`_safe_unlink`) on every path, including the idempotent-replay return and all error branches.
Backend RAM during an import is now bounded to one 1 MiB chunk instead of up to the full upload. `agent_http_request` retries only on `ConnectError` (pre-body), so the handle's read position is never consumed before a retry.

**L1 — fixed.** `materialize_data_paths` (`services/git_service.py`) now validates each declared entry against `_is_safe_data_path` (`_SAFE_DATA_PATH_RE`: alphanumerics, path separators, and glob metacharacters only — `re.fullmatch`, so an embedded/trailing newline is rejected). Entries carrying shell metacharacters (`"`, `` ` ``, `$`, `;`, `|`, `&`, `\`, `<`, `>`, …) are dropped with a `logger.warning` before any container write; an all-unsafe declaration is a clean no-op. This keeps a malformed/malicious template from corrupting the heredoc write (which already crossed no privilege boundary — it runs in the agent's own container).

**I1 — accepted as-is** (consistent with existing owner-operation MCP tools; no change).

**Tests** (TDD, all green — 43 passed): `test_data_paths_allowlist.py` adds `test_materialize_data_paths_drops_shell_metachar_entries`, `…_all_unsafe_is_noop`, `test_is_safe_data_path_charset`; `test_agent_data_endpoints.py` adds `test_import_413_when_content_length_exceeds_cap`, `test_import_413_from_stream_cap_cleans_up`, `test_import_cleans_up_staged_temp_on_success`, `test_drain_upload_to_file_enforces_cap`, `test_drain_upload_to_file_returns_size_under_cap`.
