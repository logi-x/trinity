## OWASP / verification trace

- **A01 Broken Access Control — PASS.** `GET`/`PUT /max-parallel-tasks-ceiling` both call `require_admin(current_user)` (established raising gate, used 8× in the same file). `PUT /{name}/capacity` keeps the owner gate `db.can_user_share_agent(...)` → 403 otherwise. The generic catch-all `PUT /{key}` **explicitly rejects** `max_parallel_tasks_ceiling` with 422, so the range-validation can't be bypassed by writing the raw key (mirrors the `skills_library_url` SSRF special-case). Per-agent upper bound is read from the ceiling, not a hardcoded 10 — no privilege to exceed it.
- **A03 Injection — PASS.** Value persisted via `db.set_setting(KEY, str(body.value))` where `body.value` is a Pydantic `int` re-validated to 1–32 before the write. No string interpolation into SQL or shell.
- **A04 Insecure Design — PASS.** Range enforced server-side (`MIN ≤ value ≤ MAX`, 1–32) with descriptive 400; client cannot widen the bound. Per-agent endpoint changed from untyped `body: dict` to `AgentCapacityUpdate` (missing field → 422).

## Fail-open analysis (explicitly reviewed, not a finding)

`get_max_parallel_tasks_ceiling()` and the canary `_clamp_to_ceiling()` fail **open to the conservative default (10)**, not to unbounded, on a settings/DB read failure. The only thing weakened on failure is a *resource-control* limit (a hard-exclusion DoS category), and it degrades to a bounded default rather than removing the cap. The no-cache read-through is a deliberate `--workers 2` correctness choice (a TTL cache could let a stale worker admit above a just-lowered ceiling). Acceptable by design.

## Integrity invariants (canary) — preserved

- **S-02 (no overbooking)** intentionally keeps the **stored** cap as the upper bound. Correct: the runtime clamp only ever *lowers* the effective admit limit vs stored, so `ZCARD` can never exceed stored — a real `acquire_slot` concurrency bypass is still caught. Detector not weakened.
- **B-02 (no queued without slots full)** switched to the **effective** cap to avoid a false-fire when an agent is effective-full under a lowered ceiling. Snapshot carries `effective_max_parallel` (None → falls back to stored for pre-ceiling test constructors).

## Known limitation (documented in-code, accepted)

`agent_call_limiter` freezes its per-agent semaphore cap at first access (`_AGENT_SEMAPHORE_CAPS`); a live agent's semaphore does not shrink on a ceiling/cap drop until process restart. New agents get the clamped cap immediately. This is a capacity-tightening lag, not a security boundary — no over-privilege results (worst case an already-running agent keeps its prior, higher concurrency briefly). Out of scope per the PR.
