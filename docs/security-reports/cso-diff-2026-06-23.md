## Architecture mental model (Phase 0)

First-time setup is an intentionally **unauthenticated** flow (`/api/setup/status`, `/api/setup/admin-password`) that must work on a fresh install with no users yet. It **self-disables** the moment the admin account is created (`setup_completed` flag → 403 forever after).

**This change** removes the single-use **setup token** (#1165 / SEC #177) that previously gated `POST /admin-password`. The token was a Redis-shared secret printed only to the server logs, so a purely-remote attacker (no host/log access) could not complete setup. The diff also makes **admin email required** and rebuilds the setup UI. Per the operator's explicit decision earlier in this work item, the security tradeoff was chosen as **"accept + document"** (drop the token everywhere; document the operator's responsibility to network-restrict the pre-setup window).

### [F1] Pre-setup admin-account hijack window — setup token removed (HIGH · 10/10 · VERIFIED · **ACCEPTED, documented decision**)

**File**: `src/backend/routers/setup.py` — `POST /admin-password` (no auth dependency; only the `setup_completed` self-disable gate)

**Evidence**:
```python
@router.post("/admin-password")
async def set_admin_password(data, request, background_tasks):
    if db.get_setting_value('setup_completed', 'false') == 'true':
        raise HTTPException(status_code=403, ...)   # self-disable gate
    # ... no setup token, no auth — validates password + email, then creates admin
```
Verified live against the running backend: an unauthenticated POST reaches the handler and is processed (rejected only on input validation — 422 missing email, 400 bad email/weak password/mismatch). A *valid* unauthenticated POST would create the admin account.

**Exploit scenario**:
1. Operator deploys Trinity on an internet-reachable host and has not yet completed setup.
2. An attacker reaches `POST /api/setup/admin-password` before the operator does.
3. Attacker submits a valid email + strong password → 200, admin account created with attacker-controlled credentials, `setup_completed=true`.
4. The legitimate operator is locked out; the attacker holds full admin (create agents, read injected credentials, fleet control).

**Why this is a regression vs the prior design**: the removed setup token required reading the server logs (host/log access), so a *purely remote* attacker could not complete setup. Without it, the only barrier during the pre-setup window is network reachability.

**Status — ACCEPTED, documented decision (not an oversight)**: the operator explicitly chose "drop the token everywhere + document the tunnel/VPN guidance" for this work item, to remove friction from the common self-hosted, single-operator bring-up. Mitigations now in place:
- **Documented operator responsibility** — `docs/DEPLOYMENT.md` → Security Recommendations (item 1): keep an internet-reachable instance behind a tunnel/VPN until setup completes. Mirrored in `routers/setup.py` docstrings, the `main.py` startup log, `architecture.md`, and `requirements.md` §15.2/§43.2.
- **Self-disabling window** — exposure is limited to the pre-setup window only; after the admin account exists, all auth is enforced.
- **Non-issue on localhost / trusted LAN** — the common dev/eval case.

**Residual-risk options (if the team later wants defense-in-depth without the log round-trip)** — not required by this PR:
- Bind the localhost-skip variant (auto-skip token only for loopback/RFC1918 clients, keep it for public IPs) — the "skip on local, keep on remote" option originally weighed.
- Replace the token with a host marker file (proof-of-control).

## Categories checked clean (diff scope)

- **Secrets (P2)**: no credentials introduced in the diff (token removal *reduces* secrets-in-logs); only CSS `mask-image` false-positives matched.
- **A02 Crypto**: password hashing unchanged (`hash_password`/bcrypt); no new crypto. The removed `secrets.compare_digest` was the token comparison — no remaining secret to compare, so no timing surface.
- **A03 Injection**: no raw SQL — only `db.update_user*` / `set_setting` (parameterized in the db layer); no `subprocess`/`os.system`.
- **A04 Insecure Design**: password complexity still fully enforced (`validate_password_strength` + generic `PASSWORD_REQUIREMENTS_MESSAGE`, no per-rule disclosure on the unauthenticated endpoint); email validated before any write (no half-completed setup).
- **A07 AuthN**: `admin`+password and email-as-identity both work; no change to login/JWT. Email binding write wrapped so it can't block setup, logs only `type(e).__name__`.
- **A01 elsewhere**: no *new* unauthenticated endpoint added — both setup routes were already unauthenticated by necessity; this removes a control on an existing one (F1), it does not expose new ones.
- **Frontend (P7/A03)**: no `v-html` of user content; the only embedded SVG is a static brand mark; axios call wrapped in try/catch with error + loading states.
- **LLM/AI (P7)**: no agent/MCP/prompt surface touched.
