## What was checked and cleared

**1. Shell write-back injection (`start.sh`) ‚Äî VERIFIED SAFE.**
`sed -i.bak "s/^DOCKER_GID=$/DOCKER_GID=${detected}/" .env`. `${detected}` is
numeric-only on both source paths: the probe pipes through `tr -dc '0-9'`, and
the Linux fallback takes the numeric GID field (`getent group docker | cut -d:
-f3`). No metacharacters can reach `sed`. The explicit-override guard
(`grep -qE '^DOCKER_GID=[0-9]+'`) short-circuits before any probe/write.

**2. `DOCKER_GID=0` (root group) on Docker Desktop / Colima / rootless ‚Äî NOT an escalation.**
The probe now correctly returns `0` where the socket is `root:root`-owned, so the
backend joins supplementary group 0 to reach it. This is *required* to access a
root-group-owned socket as a non-root (UID 1000) process ‚Äî there is no
lower-privilege alternative. It is not additional privilege beyond what Docker
socket access already implies (root-equivalent by design, Invariant #11). Native
Linux prod is unchanged: the probe returns the non-zero host `docker` GID
(999/991/990), exactly as before.

**3. Probe mounts the docker socket into a throwaway `alpine` (unpinned) ‚Äî operator-scope, excluded.**
`docker run --rm -v /var/run/docker.sock:... alpine stat ...` runs only inside the
operator's `start.sh`; it is not attacker-reachable, and the operator already
trusts the daemon and every image `compose up` pulls. Falls under the
missing-hardening exclusion. *Optional hardening:* pin `alpine` by digest to
shrink the deploy-time supply-chain surface.

**4. Logging change (`docker_service.py`) ‚Äî net security improvement.**
The swallowed Docker exception (`PermissionError` / `docker.errors.APIError`) now
reaches the Vector-captured structured log instead of `print()`. The message
carries socket/permission text only ‚Äî no credentials or env. The module-level
throttle counter is a benign log-rate guard (race-safe per the GIL note); not a
security concern.

**5. New CI job (`verify-docker-gid-detection`) ‚Äî clean.**
Runs on `pull_request` (**not** `pull_request_target`) under the workflow's
top-level `permissions:` block, so fork PRs get a read-only token and no secrets.
The step is a hermetic shimmed bash test with no `${{ github.event.* }}`
interpolation in `run:`. `actions/checkout@v6` is first-party/unpinned (MEDIUM at
most per FP rules) and matches the existing convention across the whole
workflow ‚Äî not a regression introduced here.
