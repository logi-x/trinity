## Detailed Review

### Secrets тАФ CLEAR
No hardcoded API keys, tokens, PEM blocks, or generic secret assignments in the
additions. No `.env` files in the diff. No dependency manifest changes (no new
packages to review).

### Auth boundaries тАФ CLEAR (the load-bearing property)
The new `resolve_and_validate_execution(execution_id, agent_name)` is the auth
gate for the effect scope. `agent_name` in every sink comes from an
**authenticated** source тАФ `AuthorizedAgentByName` (messages, voip) or
`can_user_share_agent` + the agent-scoped-key match (share_file) тАФ never from
agent-supplied data. The agent-supplied `execution_id` is validated against that
authenticated `agent_name`: an execution belonging to another agent тЖТ mismatch тЖТ
`None` тЖТ **fail-open (no claim, no replay)**. Therefore a forged/borrowed
`execution_id` **cannot** read another tenant's stored snapshot and **cannot**
send as another agent (the effect always runs as the authenticated caller). The
validation runs *before* the claim, closing the cross-tenant read path.

Nevermined settle scope is keyed on `agent_request_id`, which comes from the
Nevermined-server-trusted verify result (not agent input), so a payment scope
can't be forged. The terminal-turn no-settle guard is preserved.

### Injection тАФ CLEAR
`identifying_args` and `dedup_label` (both agent-supplied) are only ever
`json.dumps`-canonicalized and fed to `hashlib.sha256` to derive the key тАФ never
interpolated into SQL, a shell, a template, or `eval`. No `os.system` /
`subprocess` / `eval` / SQL-string-interpolation in the additions. No `db/`
files changed: the feature reuses the existing parameterized SQLAlchemy
idempotency ops (`db/idempotency.py`).

### Platform patterns тАФ CLEAR (1 LOW noted above)
Snapshots stored for replay are sanitized JSON dicts (`message_id`,
`call_id`+`status`, settle receipt `{success,payer,credits,tx_hash}`, share URL),
never raw provider objects and never a payment credential (the x402
`access_token` / payment-signature is deliberately NOT stored). `share_file`
keeps the existing `validate_publish_path` traversal guard; the new content
sha256 is computed over already-extracted bytes (no new path surface). The
in-flight-duplicate path raises `EffectInProgressError` тЖТ 409 with a generic
message (no info leak), never a silent skip-and-succeed.

### Configuration тАФ CLEAR
No new env vars, no CORS/debug/cookie/header changes. Fail-open is deliberate,
logged (`logger.warning`), and consistent with the established #525 dedup
posture (the dedup layer must never block a real send); it is not a silent
exception swallow.
