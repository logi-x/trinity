## Architecture summary

#846 adds an owner opt-in flag (`agent_ownership.mcp_exposed`). When set, the Trinity
MCP server polls a new internal endpoint (`GET /api/internal/mcp-exposed-agents`,
X-Internal-Secret gated) every ~20s and dynamically registers a dedicated
`chat_with_<slug>` tool per exposed agent. The dedicated tool binds the agent name
**server-side** and delegates verbatim to the shared `runAgentChat` body, so routing,
idempotency, and the `checkAgentAccess` execution gate are identical to
`chat_with_agent`. Tool name is a deterministic, collision-resolved slug computed by
the backend (single source of truth). Dual-track migration (SQLite + Alembic 0009).

## What was checked and is clean

- **Internal poll endpoint auth** — `/api/internal/mcp-exposed-agents` inherits the
  router-level `Depends(verify_internal_secret)` (constant-time `hmac.compare_digest`).
  MCP server warns + disables the reconciler when `INTERNAL_API_SECRET` is unset.
- **Execution access gate** — `runAgentChat` calls `checkAgentAccess(apiClient, auth,
  agent_name)` with a server-bound name; unauthorized callers get an "Access denied"
  envelope. No bypass.
- **Injection** — tool name is a computed `[a-z0-9_]` slug; DB writes go through
  SQLAlchemy with static DDL. No SQL/command injection.
- **Frontend XSS** — `McpExposedPanel.vue` uses auto-escaped `{{ }}` only (no `v-html`);
  rendered `tool_name` is the constrained slug.
- **Authz separation** — `GET .../mcp-exposed` = `AuthorizedAgentByName` (read);
  `PUT .../mcp-exposed` = `OwnedAgentByName` (owner-only) + refuses the system agent +
  audit-logs (`CONFIGURATION/mcp_exposed_config`).
- **Soft-delete guard** — getter/setter both `WHERE deleted_at IS NULL` (the mixin
  docstring explicitly notes it copied the circuit-breaker pattern, not file_sharing,
  for this reason).
- **Secrets** — no credentials in the diff.

## Resolution (2026-06-29, after Codex cross-review)

Codex (independent second voice) agreed on Finding 1's behavior (rated P2, same
intent-dependent caveat) and independently re-raised the template-label leak as a
"missed P2 that compounds Finding 1." The item **both reviewers independently flagged**
— the `trinity.template` label embedded in the globally-advertised tool description —
was resolved:

- `build_tool_description` is now **name-only** (drops the template/repo label);
  signature is single-arg so no metadata can be smuggled into the global description.
- `GET /api/internal/mcp-exposed-agents` no longer fetches Docker template labels
  (removes the leak source AND an unnecessary Docker call per ~20s poll).
- Test `test_mcp_tool_names.py` updated to lock the name-only contract.

This **fully resolves Finding 2** (no template text reaches the description → no
injection surface) and **removes the cross-tenant template/repo-identifier leak portion
of Finding 1**.

**Visibility decision — RESOLVED as Option A (fleet-wide, by design), 2026-06-29.**
Owner decided that `canAccess = connectorDenied` (every non-connector key sees the
now-name-only `chat_with_<slug>` tool) is the intended **publish** semantics, justified
by Trinity's single-tenant / sovereign deployment model — cross-key name enumeration is
not a cross-tenant leak when one trusted operator runs the instance. Execution stays
gated by `checkAgentAccess`. **Option B** (scope `canAccess` to actual agent access,
mirroring `list_agents`) was explicitly considered and declined; revisit only if Trinity
ships multi-tenant instances where agent names are sensitive. Rationale to be recorded as
a PR review comment (separate from the description). No code change for this decision —
the only Finding-1 change is the template-label scrub above, which closes the sensitive
half regardless.

### Codex-only observations (surfaced, not auto-fixed — not mutual-agreement items)
- **Slug split-brain on collision (correctness, P2):** the per-agent `GET/PUT` returns
  `compute_tool_name(name)` (no collision suffix) while the poll uses
  `compute_tool_names(full_set)` (adds suffix), so on a base-slug collision the UI shows
  a tool name that differs from the one actually registered. Already documented as a
  known limitation in `mcp_tool_names.py` ("the internal poll endpoint is authoritative").
- **Reconciler robustness (availability, P2):** `startExposedToolsReconciler` has no
  per-tool `try/catch` and no duplicate-name guard on the *desired* set, so a thrown
  `registerDynamicTool` mid-pass leaves `current` partially mutated. Backend
  `compute_tool_names` guarantees uniqueness, so this needs a backend bug to trigger.
- **Notification call shape (UX):** `McpExposedPanel.vue` calls `showNotification({type,
  message})` but `useNotification()` expects positional `(message, type)`. Real, but a
  pre-existing pattern copied verbatim from the shipped `FileSharingPanel.vue` (same bug)
  — a codebase-wide panel issue, not introduced by #846.

### Finding 1 — Exposed-agent metadata enumerable by any authenticated MCP key (MEDIUM, VERIFIED)

**Where**: `src/mcp-server/src/tools/dynamic-agents.ts:248` (registers each dedicated
tool with `connectorDenied` as its `canAccess`), `src/mcp-server/src/server.ts:241`
(`connectorDenied = (auth) => auth?.scope !== "connector"`).

**Behavior (verified end-to-end)**: FastMCP builds each session's tool set as
`this.#tools.filter(t => t.canAccess ? t.canAccess(auth) : true)`
(`node_modules/fastmcp/.../chunk-LWU5CQGW.js:1731` and the post-start add filter at
:550). Dedicated tools pass only `connectorDenied`, which is `true` for **every**
non-connector scope (`user`, `agent`, `system`). There is no per-agent access check on
visibility. So any authenticated MCP key — including an `agent`-scoped key from a
different tenant that `agent_permissions` would otherwise hide — sees, in its `tools/list`,
a `chat_with_<slug>` entry for **every** exposed agent, with a description of
`Chat directly with the "<agent_name>" agent. Template: <trinity.template>.`

Execution is NOT bypassed: a call still hits `checkAgentAccess` and returns "Access
denied". The leak is **metadata/enumeration only**.

**Why it matters**: This regresses an existing security property. `list_agents` (and
the per-agent permission model) deliberately returns *only permitted agents + self* —
an unauthorized user cannot enumerate agent names they lack access to. #846 makes
exposed agents' **names and template labels** visible to all non-connector keys. Template
labels routinely carry repo identifiers (`github:Org/private-repo`), so a private repo
name can leak to users/tenants who can't access the agent. The panel's own copy —
"Access is unchanged: callers still need ownership or a share to actually chat" — frames
exposure to the owner as execution-only, not "broadcast this agent's name + template to
every MCP user."

**Exploit scenario**:
1. Tenant A's owner toggles `mcp_exposed` on `acme-finance-bot` (template
   `github:AcmeCorp/finance-internal`).
2. Tenant B holds an ordinary `user`- or `agent`-scoped MCP key with zero access to that
   agent.
3. Tenant B issues `tools/list`. The response contains
   `chat_with_acme_finance_bot` → "...Template: github:AcmeCorp/finance-internal."
4. Tenant B has now learned the agent's existence, name, and private repo path without
   any access grant. (Calling the tool returns Access denied — the disclosure is the
   list entry itself.)

**Likely intent caveat**: the feature is named "publish a surface," so some visibility
is intended. The decision to confirm is *to whom*: today it is **everyone except
connector keys**, with no per-access scoping and with template labels included.

**Recommendation** (pick one, in order of preference):
1. Scope dedicated-tool `canAccess` to actual agent access, not just `connectorDenied` —
   e.g. `canAccess: (auth) => connectorDenied(auth) && callerCanAccessAgent(auth, name)`,
   mirroring how `list_agents` filters. This keeps execution AND visibility consistent.
   (Cost: `canAccess` is sync; an access check may need a cached/precomputed access set
   per session rather than a live backend call.)
2. If global visibility is intended, **drop the template label from
   `build_tool_description`** (name-only description) so no repo/template metadata leaks
   to non-owners.
3. At minimum, document the global-visibility semantics in the panel copy so owners
   opt in knowingly.

### Finding 2 — Template label flows unsanitized into advertised tool description (LOW, TENTATIVE)

**Where**: `src/backend/services/agent_service/mcp_tool_names.py:79-82` —
`desc += f" Template: {template_label}."`, sourced from the `trinity.template` Docker
label and advertised as the MCP tool description to every connected LLM client.

**Concern**: tool descriptions are LLM-visible context. If an actor can place freeform
text (newlines / instruction-like content) into the template label of an agent they then
expose, that text reaches the tool-description context of all connected clients — a
second-order prompt-injection surface.

**Why TENTATIVE / not promoted**: typical template labels are constrained
(`github:Org/repo`, validated local template names) with little freeform room, so a
clean injection path is not yet demonstrated. Worth confirming the full set of values
that can land in `trinity.template` (custom/local templates, rename flows) before
treating it as actionable. Folds into Finding 1's recommendation #2 (name-only
description removes the vector entirely).
