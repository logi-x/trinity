## Executive Summary

Trinity remains **well-engineered on most security axes** — SQL is uniformly parameterized, no `shell=True`/`os.system` anywhere in the backend, AES-256-GCM credential encryption is textbook, JWT is algorithm-pinned, the two-Docker-network split correctly keeps agents off Redis, prod images are non-root, and the MCP agent-to-agent permission model is enforced server-side with no spoofable inputs.

However, this audit surfaces **one CRITICAL architectural finding** that undercuts the platform's central isolation promise:

> **The per-agent internal HTTP server (`agent-server`) has no authentication and is directly reachable by every other agent on the shared `trinity-agent-network`.** A single compromised or prompt-injected agent can read every co-resident agent's plaintext credentials (`.env` — external API keys, OAuth tokens, and the victim's own Trinity MCP key), drive arbitrary executions on them, and tamper with their files. The whole `agent_permissions` model (enforced only at the MCP layer) is bypassed by talking HTTP to a peer's port 8000 directly.

> ✅ **Remediated on `dev` by #1159** (see banner above) — `AgentAuthMiddleware` now enforces a per-agent `X-Trinity-Agent-Token` on every agent-server route. The finding stands as audited on `main`; it is closed in v0.7.0.

This is the headline. The remaining findings are dependency hygiene (2× HIGH, trivially fixable) and a set of MEDIUM/LOW hardening gaps, several of which are defense-in-depth layers *underneath* the CRITICAL.

## F1 — CRITICAL — Unauthenticated agent-server is peer-reachable across the agent network

**Confidence 9/10 · VERIFIED end-to-end · NEW (not in any prior report)**

> ✅ **REMEDIATED on `dev` by #1159** (commit `0786f0b4`), shipping in v0.7.0. `AgentAuthMiddleware` (`docker/base-image/agent_server/main.py`) now enforces a per-agent `X-Trinity-Agent-Token` = `HMAC-SHA256(AGENT_AUTH_SECRET, "trinity-agent-auth:v1:"+name)` on **every** agent-server route (HTTP + WS) except `/health`, and the permissive CORS (`allow_origins=["*"]`) was removed. The token is derived (not stored) and the master secret lives only in backend env, so a compromised agent holds its own token but cannot compute a peer's — the cross-agent exploit below is closed. The finding text is preserved as audited on `main` (pre-cut).

### What
Every Trinity agent runs an internal FastAPI server (`docker/base-image/agent_server/`) on `0.0.0.0:8000`. Its own docstring asserts *"This server is NOT exposed externally. All access goes through the authenticated Trinity backend."* — but **nothing enforces "backend only."** Three facts, each independently verified, compose into a critical breach:

1. **No authentication on any endpoint.** `main.py:46-52` installs only a permissive CORS middleware (`allow_origins=["*"]`); no router declares `Depends(...)`, `Security(...)`, or any shared-secret check (grep across `credentials.py`, `files.py`, `chat.py` = empty). `run_server()` binds `host="0.0.0.0"` (`main.py:100`).
2. **The backend itself sends no secret.** `services/agent_client.py` issues plain `http://agent-{name}:8000` requests with no `Authorization`/`X-Internal-Secret` header — so there is *no secret a peer attacker would lack*.
3. **All agents share one flat network with inter-container communication enabled.** `docker-compose.yml:415-420` / `docker-compose.prod.yml:429-434` define `trinity-agent-network` as a plain `driver: bridge` with only a `subnet` — no `internal: true`, no `com.docker.network.bridge.enable_icc=false`, no `driver_opts`. Docker's default ICC is on, and Docker DNS resolves sibling `agent-<name>` names. Agent creation hard-codes this network (`services/agent_service/crud.py:736`, `lifecycle.py:549`, `system_agent_service.py:241`).

The crown-jewel endpoint is unauthenticated and returns plaintext secrets:

```python
# docker/base-image/agent_server/routers/credentials.py:183
@router.get("/api/credentials/read")
async def read_credential_files(paths: str = Query(...)):
    ...
        content = filepath.read_text()
        files[rel_path] = content      # returns full .env / .mcp.json contents
```

### Exploit scenario
1. Agent **A** is compromised — by prompt injection (it processes an attacker-authored email / web page / document — the documented industry risk Trinity ships guardrails for), or it is simply a malicious agent a low-trust user was allowed to create. Either way it has a shell (Claude Code runs bash).
2. A enumerates peers: Docker DNS (`agent-<guessed-name>`) or a sweep of `172.28.0.0/16` for open `:8000`.
3. A runs, against victim agent **B**:
   ```
   curl 'http://agent-B:8000/api/credentials/read?paths=.env,.mcp.json'
   ```
   and receives B's entire `.env` — every external service credential (Google/Slack/GitHub PAT/Notion OAuth tokens, third-party API keys) **and** B's own `TRINITY_MCP_API_KEY`.
4. With B's MCP key, A now also acts *as B* in the MCP layer (lateral movement that the `agent_permissions` model was meant to prevent). A can likewise `POST http://agent-B:8000/api/chat` to run arbitrary instructions on B, or `PUT /api/files` / `POST /api/credentials/update` to tamper with B's workspace and credentials.

The same applies to `GET /api/files/download?path=/home/developer/.env` (read) — credential exfiltration has two independent paths.

### Impact
Complete failure of agent-to-agent isolation: cross-agent credential exfiltration, execution-as-peer, and file/credential tampering, for **every** co-resident agent. Blast radius from one compromised agent = the secrets of the entire fleet. This defeats the platform's restrictive-default `agent_permissions` model, which only governs MCP calls.

### Why it scored CRITICAL (and the one honest caveat)
The precondition is "control one agent first" — but running autonomous, instruction-following agents *is* Trinity's purpose, and prompt-injection compromise of one agent is a realistic, in-scope threat the platform explicitly defends against elsewhere. In a single-operator deployment this is a blast-radius/lateral-movement issue; in any multi-user/shared deployment it is a direct tenant-isolation breach. Either way, an unauthenticated network-reachable endpoint that hands out another principal's plaintext credentials is critical. The 1 withheld confidence point reflects the precondition + peer-discovery step, both realistic.

### Remediation (defense in depth — do more than one)
1. **Authenticate the agent-server.** Inject a per-agent shared secret (or mTLS) that only the backend holds; require it on every agent-server route via a router-level dependency. The backend already injects `TRINITY_MCP_API_KEY`; add an analogous backend→agent secret and validate it. This is the real fix.
2. **Network-segment agents from each other.** A flat shared network is the root enabler. Options: per-agent networks (agent ↔ backend/mcp-server only), or an L3/iptables policy blocking agent→agent `:8000` while preserving agent→mcp-server:8080 and agent→otel:4317. (Note `enable_icc=false` alone is insufficient — agents legitimately need mcp-server/otel, which also sit on this network.)
3. **Bind sensitive endpoints to the backend's source identity** at minimum, even before (1) lands.
4. Add a canary/test asserting an agent container *cannot* reach a peer's `:8000` (mirror of the existing `test_agent_network_container_cannot_reach_redis`).

## F2 / F3 — HIGH — Runtime-reachable dependency vulnerabilities

> ✅ **REMEDIATED on `dev`**, shipping in v0.7.0. F2: hono bumped (#1255), undici bumped (#1289), `fastmcp` now `^4.3.0`. F3: `form-data` 4.0.5→4.0.6 (#1254), frontend `axios` `^1.18.0`. Finding text preserved as audited on `main` (pre-cut); re-run `npm audit` post-merge to confirm 0 HIGH on prod deps.

**F2 (MCP server, conf 8/10):** `fastmcp@3.32.0` is a **runtime/prod** dependency of `src/mcp-server/` and transitively pulls three HIGH advisories — `hono` (CORS-credential reflection, JWT-middleware scheme acceptance, static path traversal), `undici` (TLS cert-validation bypass, Set-Cookie injection, cache poisoning), and `form-data` (CRLF injection). `fastmcp` is the MCP server's HTTP runtime, so hono/undici are on a live path. `fixAvailable: true`.

*Calibration:* confidence the CVEs are present in a prod-reachable tree is high; real-world exploitability is conditional on how fastmcp/axios exercise the vulnerable code. Reported because dependency hygiene is squarely in scope and the fix is one bump.

**Remediation:** `npm audit fix` (or bump `fastmcp`/`axios`) in `src/mcp-server/` and `src/frontend/`; re-run `npm audit` to confirm 0 HIGH on prod deps. Dev-only `vite`/`esbuild`/`qs` advisories are below the bar (not shipped to prod).

## F4 / F5 / F8 — Agent-server file/credential boundary defects (under F1)

> 🔒 **Exploit path closed by #1159.** With the agent-server now authenticated, these endpoints are no longer peer-reachable without the per-agent token, so the cross-agent attack vector is gone. The underlying validator hardening (deny-list parity, `is_relative_to`, `..` guard) remains worth landing as defense-in-depth and is tracked as a follow-up.

These are real and worth fixing on their own, and they sharpen F1 (they govern *what* a peer attacker can write/read once it reaches the unauthenticated server).

**F4 — MED (conf 9/10): deny-list parity violated.** `src/backend/services/agent_service/files.py:23-49` carries `_FILE_WRITE_DENY_PATTERNS` with an explicit *"KEEP IN SYNC with … agent_server/routers/files.py::EDIT_PROTECTED_PATHS"* comment — but the agent-server list (`files.py:178-186`) is a strict subset, omitting `.ssh/*`, `.aws/*`, `.gcp/*`, and `.claude/settings*.json`. Since the agent-server is the *actual* boundary for direct (peer) callers, a write to `~/.ssh/authorized_keys` or `~/.aws/credentials` — blocked at the backend proxy — is **allowed** at the agent-server. Writing `authorized_keys` is RCE-adjacent. *Fix:* make `EDIT_PROTECTED_PATHS` a true superset-or-equal of the backend list (add `.ssh/*`, `.aws/*`, `.gcp/*`, `.claude/settings*.json`), ideally a single shared glob validator used by every handler.

**F5 — MED (conf 9/10): sibling-prefix confinement bypass.** Five handlers (`list_files:40`, `download_file:131`, `delete_file:270`, `preview_file:331`, `update_file:393`) gate with `str(requested_path).startswith(str(Path("/home/developer")))` — no trailing slash, base not `.resolve()`d — so `/home/developer-evil/x` passes. `../` traversal *is* separately caught by `.resolve()`. The `create_folder` handler (`:449`) was already fixed to `is_relative_to()` and its comment documents this exact bypass class — the other five were never updated. *Current impact is bounded* (writes to a new sibling EPERM-fail under root-owned `/home`/UID-1000; no sibling read targets are provisioned), so it's a latent correctness/invariant defect rather than a live breach — but it should track the already-proven fix. *Fix:* replace all five with `requested_path.is_relative_to(Path("/home/developer").resolve())`.

**F8 — LOW (conf 8/10): `/api/credentials/update` missing traversal guard.** The `files` write loop (`credentials.py:105`) does only `clean_path = file_path.lstrip("/")` then writes `home_dir / clean_path` — no `..` rejection, no containment check — while the sibling `read_credential_files` (`:202`) explicitly blocks `".."`. `file_path="../foo"` escapes lexically (bounded by root-owned FS perms). *Fix:* copy the `..` + `is_relative_to` guard already present in the read path.

## F6 — MEDIUM — CI/CD exposure

**Conf 9/10.** Two issues:
- **Secrets in PR-triggered jobs that run PR-modifiable code.** `frontend-e2e.yml:52,68` and `container-security.yml:68,83` pass `E2E_ADMIN_PASSWORD` / `E2E_ANTHROPIC_API_KEY` into `pull_request`-triggered jobs that execute `start.sh` and Dockerfiles. Safe under GitHub's default fork-secret policy (forks get no secrets without maintainer opt-in; the password is `::add-mask::`'d), but a malicious same-repo branch or an enabled "send secrets to fork PRs" setting would leak them. *Fix:* gate secret-bearing steps on `github.event.pull_request.head.repo.full_name == github.repository`; confirm the fork-secrets org setting is off.
- **First-party actions on mutable tags** (`actions/*@v6`, `github/codeql-action@v9`, etc.) — per rubric, MEDIUM. *Fix:* SHA-pin (Dependabot maintains) and add `permissions:` blocks (least privilege) to workflows that lack them, notably `deploy-dev.yml` (holds `TAILSCALE_AUTH_KEY`/`DEV_SSH_KEY`).
- *Housekeeping:* `.github/CODEOWNERS` covers `/.github/workflows/` but **not** `scripts/deploy/start.sh` or `docker/` — files those secret-bearing workflows execute. Add them, and add a second owner (current single owner is a bus-factor/bypass point).

**No CRITICAL CI findings:** no `pull_request_target`, no `${{ github.event.* }}`→`run:` script injection, all third-party actions SHA-pinned.

## F7 / F9 / F10 / F11 / F12 / F13 — LOW (verified, batch remediation)

- **F7 — IDOR on config GETs (conf 10/10).** `GET /{agent}/capabilities` (`agent_config.py:262`), `/timeout` (`:453`), `/guardrails` (`:611`) only `Depends(get_current_user)` and never call `can_user_access_agent` — unlike the `resources`/`capacity`/`api-key` GETs in the same file. Any authenticated user (role `user` suffices) can read any agent's capability flags, execution timeout, and guardrail deny-lists, and probe agent-name existence (200 vs 404). Data is non-secret operational config (no creds/PII) → LOW, but a clear Invariant #8 violation. *Fix:* switch the three signatures to `AuthorizedAgentByName` (one line each).
- **F9 — Slack verify code (conf 10/10).** `slack_adapter.py:504` uses `random.choices(string.digits, k=6)` (Mersenne Twister) where `db/email_auth.py:159` correctly uses `secrets.randbelow`. The verify path has no attempt cap (wrong code just re-prompts, same code persists). MT-state recovery is infeasible (codes are emailed to the victim, not the attacker); the *real* weakness is unthrottled online brute-force of a 10⁶ space within the 10-min window. *Fix:* `secrets.randbelow(1000000)` + attempt counter with lockout.
- **F10 — Slack download SSRF/token-leak (conf 9/10).** `slack_service.py:639` GETs `url` (from inbound event `url_private_download`, `slack_adapter.py:480`) with `Authorization: Bearer <bot_token>` and `follow_redirects=True`, no host allowlist — unlike the Telegram (`api.telegram.org`) and WhatsApp (`*.twilio.com`) adapters that validate host. Gated by HMAC-verified Slack events (attacker doesn't normally control `url_private`), so defense-in-depth/token-leak-on-redirect rather than open SSRF. *Fix:* allowlist `files.slack.com`/`*.slack.com`, set `follow_redirects=False`.
- **F11 — Rate-limiters fail open (conf 8/10).** `auth.py:90-92,168-172,185-187` return `True` (allow) when Redis is `None` or throws — so OTP/login brute-force protection evaporates during a Redis outage. *Fix:* fail closed for auth limiters, or back the OTP attempt counter with a DB column so it survives Redis loss.
- **F12 — SECRET_KEY placeholder accepted (conf 8/10).** `config.py:22-25` prints "CRITICAL" but **still uses** the literal placeholder if an operator sets it — yielding world-known JWT signing (full auth bypass). *Fix:* `raise` and refuse to boot on the known placeholder, matching the Redis-password `${VAR:?}` discipline.
- **F13 — Telegram `!=` compare (conf 7/10).** `telegram_webhook.py:54` compares the 256-bit secret token with `!=`. Timing attack on a 256-bit secret is infeasible (and there's a second URL-secret factor) → hygiene only. *Fix:* `hmac.compare_digest`, matching the Slack transport.

## Areas verified CLEAN (high-signal negatives)

- **SQL injection:** all dynamic SQL in `db/` uses `?` bind params; no user-controlled `ORDER BY`/sort column; table/column interpolation only from hardcoded registries.
- **Command injection:** no `shell=True`/`os.system`/`create_subprocess_shell` in backend; `subprocess`/`exec` use list args or `shlex.quote()`; agent names `sanitize_agent_name`'d.
- **Frontend XSS:** every `v-html` routes through DOMPurify (`utils/markdown.js`); explicit `DOMPurify.sanitize()` for workspace/mermaid HTML.
- **Crypto:** AES-256-GCM with fresh `os.urandom(12)` nonce per encrypt + tag verify; bcrypt password hashing; 55/55 token generations use `secrets` (sole exception = F9). JWT HS256 pinned, `algorithms=[…]` on every decode.
- **SSH:** client submits public key only — no server-side private-key generation/return.
- **Network ↔ Redis:** agents cannot reach Redis (two-network split holds, verified by existing test); Redis password mandatory; backend/scheduler ACLs `-@dangerous`.
- **base_image:** allowlisted (`validate_base_image`, fnmatch) at create *and* recreate; arbitrary images 403.
- **Setup endpoint:** single-use token (`secrets.token_urlsafe`) + `setup_completed` state guard + `compare_digest`; not hijackable post-setup.
- **Containers:** all prod images non-root; agents `cap_drop: ALL` + AppArmor + hardened tmpfs, no `privileged`, no Docker socket; backend Docker socket `:ro`.
- **Webhook signatures:** Twilio (HMAC-SHA1, unconditional, no sandbox bypass), Slack (HMAC-SHA256 + 5-min replay window + `compare_digest`), Telegram (dual-factor), WEBHOOK-001 (opaque token + idempotency + rate limit), x402 (payment verified server-side before LLM work) — all enforced.
- **VoIP WS:** single-use, scope-bound (`voip:{call_id}`), short-TTL ticket + GETDEL'd staged intent — no hijack/audio-injection.
- **MCP permission model:** auth context server-validated (not client-supplied); agent-to-agent gating enforced at MCP layer, fails closed; `operator_queue` agent-scoping correct.
- **Cross-user prompt injection:** MEM-001 memory keyed strictly by verified email, never cross-user; write path resolves email server-side from execution_id. (Self-injection is out of scope.)
- **SSRF (GitHub templates, skills library, media):** hosts are hardcoded constants or strict allowlists; WhatsApp `*.twilio.com` suffix check not bypassable.
- **Secrets archaeology:** 2,299 commits scanned — no leaked AWS/OpenAI/GitHub/Slack/NVM/Google/Twilio secrets; all hits are documented placeholders or test fixtures; no `.env`/`.pem` ever tracked.
- **Skill supply chain:** `.claude/skills/` clean — no credential-exfil-to-external-host, no prompt injection (the `IGNORE PREVIOUS`/`ANTHROPIC_API_KEY` strings in `cso/SKILL.md` are this skill's own detection ruleset).
- **A09 logging:** credential ops audit-log file names/counts only, never values.
