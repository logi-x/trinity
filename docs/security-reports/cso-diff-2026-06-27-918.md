## Findings

### F1 — `agent_reports` omitted from agent delete-cascade and rename → cross-tenant report disclosure under name reuse

| | |
|---|---|
| **Severity** | HIGH |
| **Confidence** | 9/10 — VERIFIED (exploit path traced; the repo's own parity guard fails on it) |
| **Category** | Broken Access Control (OWASP A01) |
| **Phase** | P9 / P12 |

**Evidence**
- `src/backend/db/agent_cleanup.py:96-180` — `AGENT_REFS` registry (drives
  `cascade_delete`). `agent_reports` is **absent** (`grep -c agent_reports
  agent_cleanup.py` → 0). Every other per-agent table (incl. the analogous
  `agent_compatibility_results`, `agent_notifications`) is registered CASCADE.
- `src/backend/db/agent_settings/metadata.py:90-219` — `rename_agent` uses an
  explicit per-table UPDATE list (and a matching docstring). `agent_reports` is
  not in it. (`cascade_rename` in `agent_cleanup.py:299` has **no callers**, so
  it does not cover the gap.)
- `src/backend/db/agents.py:123-140` — `is_agent_name_reserved` checks only
  `agent_ownership`; after a rename the old name is removed from
  `agent_ownership`, so it is **immediately** claimable by another user.
- `src/backend/db/agents.py:326-345` — `can_user_access_agent` grants access to
  the **owner** of the (reused) name → the new owner passes the gate.
- `src/backend/routers/reports.py:85-94,150-156` — `list_agent_reports` /
  `get_report` then return the orphaned rows; `GET /api/reports/{id}` includes
  the full decoded `payload`.
- `tests/unit/test_agent_cleanup_parity.py:118-133` —
  `test_every_schema_agent_column_is_registered_or_exempt` computes
  `unmanaged = schema_cols - registered - _EXEMPT`. `('agent_reports',
  'agent_name')` is a NOT-NULL agent column in `db/schema.py:414` and is in
  neither `AGENT_REFS` nor `_EXEMPT` → **this guard is RED on the branch.** It is
  the regression test purpose-built for exactly this bug class (#129/#834).

**Exploit scenario (rename variant — no retention mitigation)**
1. Tenant A (`user1`) owns agent `alpha`; its agent publishes reports whose
   payloads hold sensitive domain data (the feature's own comments note this is
   why the WS broadcast is deliberately thin).
2. `user1` renames `alpha` → `alpha-v2`. Report rows keep `agent_name='alpha'`
   (not in the rename list) → orphaned: invisible to `alpha-v2`, and `alpha` is
   no longer in `agent_ownership`.
3. Tenant B (`user2`, any `creator`) creates a new agent named `alpha`
   (immediately free per `is_agent_name_reserved`).
4. `user2` calls `GET /api/agents/alpha/reports` then `GET /api/reports/{id}` —
   `can_user_access_agent` returns True (owner) → `user2` reads Tenant A's
   historical report payloads. **Cross-tenant disclosure.**

A delete+recreate variant exists too, but is partially bounded by the 90-day
`agent_reports_retention_days` sweep (closes once orphans age out). That sweep
**does not** help the rename variant (recent payloads, instant reuse) and can be
disabled (`agent_reports_retention_days=0`).

**Impact** — A second tenant on a shared instance reading another tenant's
report payloads. Names like `ops`, `recon`, `assistant` are commonly reused.
Secondary: rename silently loses an agent's report history; orphan rows
accumulate after purge (violates the L-03 cascade invariant the codebase
otherwise maintains — and `agent_reports` is also not in the canary L-03 list,
so the watcher won't flag it either).

**Recommendation**
- Add `AgentRef("agent_reports", "agent_name", Policy.CASCADE)` to `AGENT_REFS`
  (`db/agent_cleanup.py`) — this fixes the delete path **and** turns the parity
  test green. If retain-after-delete is the desired product behavior instead,
  add it to `_EXEMPT` with a justifying comment (mirrors `schedule_executions`
  Policy.KEEP) — but that **keeps** the cross-tenant exposure, so it is not
  recommended without also gating reads on agent identity/epoch.
- Add `agent_reports` to the `rename_agent` UPDATE list in
  `agent_settings/metadata.py` (and its docstring) — or migrate rename to call
  `cascade_rename` so the single `AGENT_REFS` registry covers both paths.
- Add a `tests/unit/test_918_*` case asserting reports are deleted on cascade
  and renamed on `rename_agent`.

## Checked and clear (no findings)

- **SQL injection** — `db/reports.py` is SQLAlchemy Core throughout; `search`
  uses parameterized `.like()`; no string interpolation into SQL.
- **XSS** — only `ReportMarkdown.vue` uses `v-html`, via
  `renderMarkdown` → DOMPurify (H-005). `ReportTable/KpiTiles/Timeline/Json` use
  escaped `{{ }}` interpolation.
- **Spoofing / authz on create** — `POST /agents/{name}/reports` is
  `AuthorizedAgent` + a self-gate (`current_user.agent_name != name → 403`);
  MCP `report` tool resolves the agent from the auth context, never tool input.
- **WS data leak** — `_broadcast_report` carries only
  `{agent_name, report_id, report_type, created_at}` on SCOPE_ALL `/ws`; clients
  refetch payload via the access-gated REST endpoint (verified by
  `test_918_report_broadcast.py`).
- **IDOR on detail** — `get_report` 404s (not 403) on no-access; `delete_report`
  is scoped by `(agent_name, id)`.
- **Input validation** — `ReportCreate` enforces `report_type` regex, title ≤300,
  ISO-8601 period bounds + ordering; payload byte-capped to 256 KiB → 413.
- **MCP client** — `createReport` path uses `encodeURIComponent(agentName)`.
- **Migration** — parameterless DDL, idempotent, dual-track (SQLite + Alembic);
  matches `schema.py`.

## Remediation applied (CASCADE)

User chose CASCADE. Fix applied and verified:
- `db/agent_cleanup.py` — added `AgentRef("agent_reports","agent_name",Policy.CASCADE)`.
- `db/agent_settings/metadata.py` — `rename_agent` now re-keys `agent_reports`
  (import + UPDATE + docstring).
- `tests/unit/test_agent_soft_delete.py` — seed an `agent_reports` row; assert it
  survives soft-delete, is wiped on purge, and is re-keyed on rename.

Verification: `test_agent_cleanup_parity.py` (the previously-RED guard) +
`test_agent_soft_delete.py` + `test_918_agent_reports_db.py` → **20 passed**.

  verified safe — DOMPurify neutralizes the one `v-html` sink).

  this new CASCADE table is detected.

Flagged, NOT fixed here (out of #918 scope — follow-ups):
- **[P2] Live rename path is not structurally guarded.** `test_agent_cleanup_parity.py`
  checks `AGENT_REFS` against `cascade_rename` (which has no callers); the
  production rename is the hand-maintained list in `metadata.py`. A future table
  can pass parity and still orphan on rename. Pre-existing, affects all tables.
- **[P2] `nevermined_payment_log` (Policy.KEEP) cross-tenant exposure.** No
  retention → a reused agent name inherits the prior tenant's payment history via
  `GET /api/nevermined/agents/{name}/payments`. Genuine latent bug in a separate
  subsystem; `schedule_executions` KEEP is effectively closed by 90d retention <
  180d purge. Worth a tracked issue (key the rollup on owner/subscription_id, or
  block reuse of names with live KEEP rows).
- **[info] WS `agent_report` trigger** still carries `agent_name`+`report_type` on
  SCOPE_ALL `/ws` (title/payload correctly stripped). Consistent with existing
  `agent_activity` behavior — accepted metadata pattern.
