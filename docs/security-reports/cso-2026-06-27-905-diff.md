## Authorization — verified end-to-end

Layering is correct and matches architecture §5 (backend resolves agent-scoped keys to owner, does **not** apply `agent_permissions`; the A2A gate lives in the MCP layer):

- **MCP gate** (`git.ts checkAgentAccess`): `system` → allow; `user`/null → allow (defers to backend); `agent` → `{self}` or explicitly-permitted target only.
- **Backend dependency**: reads (`status`, `log`, `sync-state`) → `AuthorizedAgentByName`; `pull` → `AuthorizedAgentByName` (read+shared); mutating `sync` and destructive `reset-to-main-preserve-state` → `OwnedAgentByName`.
- A shared/permitted-but-non-owner agent-scoped key passes the MCP gate but is correctly rejected by the backend for `sync`/`reset` — i.e. "shared key gets read+pull only", as designed.

## Injection / shell safety

- `git/log?limit=` is `int`-typed at the FastAPI boundary → non-integer values are 422-rejected before any shell; `limit` reaches the agent server as a structured query param, not interpolated into a command. The MCP-side clamp (`Math.min/max/floor`, 1–100) is extra defense-in-depth.
- New `?request_id=` filter uses parameterized SQLAlchemy Core (`audit_log.c.request_id == request_id`) — no SQLi.
- `X-Request-ID` is minted via `randomUUID()` and forwarded by the MCP tool; the URL path uses `encodeURIComponent`.

## Candidate observations — both REFUTED

1. **Forged `X-Request-ID` polluting audit correlation** — *Refuted / informational.* The `add_request_id` middleware adopting an inbound `X-Request-ID` is **pre-existing** (SEC-001 / #371), not introduced here; `main.py` is unmodified on this branch. A caller can only attach a chosen correlation id to *their own* row — actor identity (`actor_user`, `actor_ip`) is server-derived, the append-only hash chain is intact, and the `?request_id=` read filter is `require_admin`. No privilege or identity impact.

2. **CRLF/header-injection via reflected `X-Request-ID`** — *Refuted / not exploitable.* h11/uvicorn reject bare CR/LF in header values on both parse (inbound) and emit (outbound, latin-1 + validation), so HTTP response splitting is not reachable.

## Note for the PR (non-blocking)

The feature now makes `request_id` a security-relevant correlation field consumed in the audit trail. The trust model is fine for today (caller can only mis-tag their own server-attributed row), but if a future change ever derives any **authorization or trust decision** from `request_id`, the verbatim-adopt-from-header behavior must be revisited. Worth a one-line comment near `_audit_git` or the middleware. Not a finding.
