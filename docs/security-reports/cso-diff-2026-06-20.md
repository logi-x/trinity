### Verification notes (auth boundary — the one security-relevant code change)

The audit's primary concern was the `chat.ts` line:
`const authContext = requireApiKey ? context?.session : undefined;` → `const authContext = context?.session;`

Verified **not** an auth weakening:

- **Production (`MCP_REQUIRE_API_KEY=true`)** — literal no-op: when `requireApiKey` is always true, `requireApiKey ? context?.session : undefined` ≡ `context?.session`. Zero behavior change. This is the documented production posture (the MCP server holds no admin credentials).
- **No-auth (`requireApiKey=false`)** — `getClient()` ignores `authContext` and returns the base client regardless (the `if (requireApiKey)` branch is false), and `checkAgentAccess()` short-circuits on its `if (!authContext) return { allowed: true }` guard exactly as it did before when `undefined` was passed. With no FastMCP authenticate callback the session carries no `scope`/`mcpApiKey`, so `usePullRouting` (`authContext?.scope === "agent"`) stays false. This mode is already unrestricted by design.
- **`usePullRouting`** only fires for `scope === "agent"` + non-self + flag-ON, i.e. a validated agent-scoped MCP key — no new unauthenticated surface.
- **Endpoint parity** — routing a sequential agent→agent call from `/chat` to `/task` does not cross an auth boundary: `chat_with_agent` (chat.py:467) and `execute_parallel_task` (chat.py:1289) carry **identical** dependencies — `Depends(get_authorized_agent)` + `Depends(get_current_user)`. `/task` is the same endpoint parallel mode already uses today.
- **Idempotency (D8)** — the route token (`chat-pull-task` vs `chat`) is folded into the derived key so a flag flip can't replay a wrong-shape snapshot across endpoints under the shared `agent:{name}` scope; `scope='user'` keys are unchanged across flag states (no in-flight invalidation on deploy). Pinned by `chat.test.ts`.

Backend deny-path fix (`idempotency_service.fail(idem)` on the two CircuitOpen branches) correctly mirrors the existing `/chat` (chat.py:242) and `CapacityFull` (1577/1699) patterns; `idem` is in scope at both new sites (defined at 1363). Pinned by `test_946_task_idempotency_on_deny.py`.

Other dimensions: no hardcoded secrets in changed source; no dependency-manifest changes; `message` flows to the same backend endpoints as today (no new injection surface); the new feature flag defaults OFF in **both** services off the **same** env key (no drift) and the `/api/settings/feature-flags` exposure is an auth-gated non-sensitive boolean.
