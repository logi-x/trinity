## What the change does (context)

Closes the long-standing "zero inbound auth on the agent server :8000" hole. Every backendâ†’agent HTTP call now carries `X-Trinity-Agent-Token = HMAC-SHA256(AGENT_AUTH_SECRET, "trinity-agent-auth:v1:"+name)`; the agent-side `AgentAuthMiddleware` verifies it. Master secret stays backend-only; each container gets only its own token (derive-don't-store). Threat model (per the code's own docstrings): a compromised/prompt-injected sibling on `trinity-agent-network` should no longer be able to read a peer's `.env` or **run arbitrary Claude code on it**.

## Findings

### HIGH

**H-1 â€” `/ws/chat` WebSocket bypasses the new auth middleware (defeats the PR's core AC)**
`docker/base-image/agent_server/routers/chat.py:461-494`

`AgentAuthMiddleware` is a `starlette.middleware.base.BaseHTTPMiddleware`. Its `dispatch()` runs **only for `scope["type"] == "http"`** â€” Starlette passes `websocket` (and `lifespan`) scopes straight to the inner app without invoking the middleware. The agent server exposes a WebSocket route `/ws/chat` that `await websocket.accept()`s with **no token check** and then calls `runtime.execute(prompt=..., continue_session=True)` â€” i.e. arbitrary Claude execution on the agent, plus it returns the full `execution_log`/`raw_messages`.

Net effect: after this PR ships and the fleet is recreated (HTTP enforcement ON), any sibling on `trinity-agent-network` can still `ws://agent-{victim}:8000/ws/chat` and run arbitrary Claude turns on the victim (which can then read the victim's own `.env` and spend its credentials). This is exactly the "run arbitrary Claude code on a sibling" attack the feature claims to prevent â€” the HTTP surface is closed, the WS surface is not.

Aggravating + mitigating facts:
- **The endpoint is dead code.** No consumer anywhere in the repo (`grep ws/chat` â†’ only the definition). The browser WS path uses the backend's ticket-gated `/api/ws/ticket` â†’ `/ws`, never the agent server. So the safest fix is **delete `/ws/chat`** (removes the surface entirely). If it must stay, gate it inside the handler: read `TRINITY_AGENT_AUTH_TOKEN`, compare the first frame / a query param / subprotocol header with `hmac.compare_digest`, and `await websocket.close(code=1008)` on mismatch (BaseHTTPMiddleware cannot do this for you).
- Requires the attacker to already be a compromised agent on the internal network (lateral movement) â€” not externally reachable. That's why this is HIGH, not CRITICAL â€” but it is HIGH because it directly negates the PR's stated security objective and is trivially exploitable from any sibling.
- **No test exercises the WS path** (the isolation suite covers HTTP 401/200/health only; no agent-server WS test exists anywhere). A regression test that connects to `/ws/chat` from a tokenless sibling and asserts the connection is refused would lock the fix.

### MEDIUM

**M-1 â€” Enforcement is not automatic; the grace path leaves running agents unprotected until recreate**
`docker/base-image/agent_server/middleware/auth.py:44-48`, `services/agent_service/lifecycle.py:265-272`

By design (non-breaking rollout), an empty `TRINITY_AGENT_AUTH_TOKEN` â†’ middleware allows everything. `check_agent_auth_token_env_matches` forces a recreate that injects the token, but **only on the next `start_agent_internal`**. A continuously-running agent that is never stopped/restarted keeps an empty token env and stays in the grace (no-auth) path indefinitely â€” so an operator who sets `AGENT_AUTH_SECRET` but never recreates the fleet gets *zero* protection while believing the feature is live. This is acceptable as a rollout mechanism but must be an explicit deploy step: **after deploying, recreate the whole fleet** (and ideally surface "agents still on grace path" somewhere). Not a code bug; a deployment/communication requirement. The backend half is correctly fail-closed (raises if `AGENT_AUTH_SECRET` unset), so there is no silent-misconfig path â€” good.

### LOW

**L-1 â€” `/health` is exempt, so siblings can read runtime telemetry without a token**
`docker/base-image/agent_server/middleware/auth.py:34,51`

Exact-path `/health` is exempt for the (HTTP) liveness probe. Per architecture.md, `/health` returns `active_tasks`, `last_task_at`, `consecutive_failures`, and #333 diagnostics gauges. Any sibling can read these unauthenticated. Low sensitivity (no secrets, no execution), and the exemption is exact-match (`/api/health`, `/`, `/health/` all require the token â€” fail-safe). Acceptable; noted for completeness.

## Positives (verified, not just asserted)

- **Constant-time compare** â€” middleware uses `hmac.compare_digest` (no token-guessing timing oracle). âś“
- **Derive-don't-store; master never leaves backend** â€” a compromised agent holds `token_self` but cannot compute `token_peer` without `AGENT_AUTH_SECRET`. Knowing your own token grants nothing beyond your own already-owned surface. âś“
- **Spoof/staleness resistance** â€” `merge_auth_headers` strips any caller-supplied `X-Trinity-Agent-Token` **case-insensitively** before stamping the derived one, on both central clients (`agent_client.py:779`, `helpers.agent_http_request:151`). âś“
- **CORS hardening** â€” removed `allow_origins=["*"] + allow_credentials=True` from the internal-only agent server (real attack-surface reduction, not just cleanup). âś“
- **Credential-sanitizer-friendly** â€” header name contains `TOKEN`, so the existing `.*TOKEN.*` redaction masks it in transcripts. âś“
- **Rename handled** â€” deterministic derivation + `check_agent_auth_token_env_matches` forces a recreate re-deriving under the new name (no permanent 401 after rename). âś“
- **HTTP completeness** â€” every backendâ†’agent HTTP call site stamps the token: all routers (a2a, agent_files, avatar, chat Ă—11, credentials, public, system_agent, voice), services (git, github_pat_propagation, monitoring, task_execution, cleanup Ă—3, credential_encryption, dashboard, metrics, stats), pooled `AgentClient.request`, and the retrying `agent_http_request`. `lifecycle.py:62` only probes the exempt `/health`. No miss found. âś“
- **Scheduler is not a gap** â€” the standalone scheduler calls only the backend's `/api/internal/*` (the old `scheduler/agent_client.py` is deleted); it never talks to agents directly, so its lack of `AGENT_AUTH_SECRET` is correct, not a hole. âś“
- **Secrets** â€” no hardcoded value; `start.sh` `openssl rand -hex 32` persist-once (same contract as `SECRET_KEY`/`INTERNAL_API_SECRET`); `.env.example` blank; forwarded to the backend service in both compose files. âś“
- **No new dependencies; no injection vectors** (HMAC over a sanitized `agent_name`; exact string compare; no SQL/command/path/template/SSRF surface introduced). âś“

## Non-security caveat (verify, not a finding)

`AgentAuthMiddleware` wraps the whole app in `BaseHTTPMiddleware`, which historically interferes with `StreamingResponse`/SSE. The agent server serves SSE (`/api/executions/{id}/stream`). Confirm SSE log streaming still flows end-to-end under the middleware (a `/verify-local` agent-exercise run covers this). Functional, not security.
