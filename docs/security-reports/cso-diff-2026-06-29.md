Issue #1376: in Session mode the turn endpoint is synchronous; an intermediary
(Cloudflare ~100s edge timeout) can sever the socket while the backend keeps
running. The fix reconciles a severed POST against authoritative server state
(`GET /api/agents/{name}/sessions/{id}`) before declaring failure, and replaces
the attempt-count poll bound with a wall-clock deadline routed to a neutral
notice.

## Phase results

- **P0 Architecture / P1 Attack surface:** No new endpoints, auth surfaces,
  file-upload points, WebSocket channels, or external integrations. The diff
  reuses the existing authenticated, **per-user-scoped** session endpoints
  (404-on-mismatch — owners cannot read other users' sessions). Zero net new
  attack surface.
- **P7 LLM/AI · P9-A03 Injection:** `error` and `notice` render via Vue text
  interpolation `{{ }}` (auto-escaped) — no `v-html`, `innerHTML`, `eval`, or
  `new Function`. The error string derived from `e.response?.data?.detail` is
  displayed escaped. No XSS sink introduced.
- **P9-A01 Access Control:** Reconciliation (`loadSession`) calls
  `axios.get('/api/agents/{name}/sessions/{id}')` with the auth header through
  the shared client (Invariant #7), reading **only the caller's own session**.
  No IDOR; success-detection counts assistant messages on the user's own row.
- **P12 FP filtering:** Nothing reaches 8/10.

## Observation (not a finding)

`window.__TRINITY_SESSION_POLL_DEADLINE_MS__` (sessions.js:347) is a documented
e2e test seam read in `startPolling`. It only shortens the **local client's own**
poll deadline — no cross-user impact, no privilege change, no server trust. An
actor able to set it already controls their own page. Falls under the
"missing-hardening without concrete vulnerability / self-affecting" exclusion;
benign, but tracked here for transparency.
