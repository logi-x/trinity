## Why nothing surfaced (candidates considered and discarded)

1. **SHA-256 as "weak/misused crypto"** — discarded. The hash is a non-security fingerprint for response-equality detection; a collision would at worst stop a loop one iteration early. No security boundary depends on it. (Phase 9 A02 FP.)
2. **No upper bound on `no_progress_threshold`** — discarded. `ge=0` with no ceiling, but `max_runs` is hard-capped 1–100, so a huge threshold merely disables detection. No resource or logic impact. (Phase 12 exclusion #4 — input validation without proven impact.)
3. **DoS angle** — discarded; the feature *reduces* runaway loop cost. (Phase 12 exclusion #1.)
4. **Injection via the new field** — no path. Integer-typed end-to-end; DB writes are parameter-bound; migration DDL is a constant. (Phase 9 A03.)
5. **Authz / IDOR** — unchanged. The field rides an already-authenticated, ownership-gated endpoint and MCP tool. (Phase 9 A01.)
