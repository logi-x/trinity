## 2026-06-23 — pitfall — Adding a column needs BOTH schema.py AND tables.py; parity test guards only one
**Context**: `agent_ownership.voice_name` (trinity-enterprise#28). `tests/unit/test_schema_parity.py` compares `db/schema.py` (raw DDL) against `db/migrations.py` (the SQLite runner) — it does **not** import `db/tables.py` (the SQLAlchemy Core MetaData that Alembic autogenerates from and that every `db/*.py` accessor binds columns against). So a change that adds a column to `schema.py` + `migrations.py` but forgets `tables.py` keeps the parity test green, yet `select(agent_ownership.c.<col>)` blows up at runtime on the first call.
**Lesson**: A schema change touches **four** files — `db/schema.py`, `db/tables.py`, a SQLite migration in `db/migrations.py`, and an Alembic revision under `db/../migrations/versions/`. Don't trust schema-parity to catch a missing `tables.py` Column. The real guard is a db-accessor unit test that executes a live `select(table.c.<new_col>)` against a migrated DB (db_harness) — it fails loudly if `tables.py` was missed.

## 2026-06-28 — pitfall — `@dataclass` on a fieldless str-Enum breaks member equality
**Context**: `TaskExecutionErrorCode(str, Enum)` is decorated `@dataclass` (`services/task_execution_service.py:75`). `@dataclass` synthesises an `__eq__` that compares the (zero) fields, so for enum members `AUTH == BILLING` — and even `TIMEOUT == AUTH` — all evaluate **True**. `apply_result`'s breaker line `envelope.error_code == TaskExecutionErrorCode.AUTH` therefore fired for *any* non-None code reaching it (504→TIMEOUT pre-existing; 429→BILLING after #1085 populated it), wrongly tripping the AUTH dispatch breaker. Found in the #1085 review; fixed by value-comparing (`error_code.value == "auth"`).
**Lesson**: Never put `@dataclass` on an `Enum` subclass. Compare enum codes by `.value` (or plain `is`/identity), never `==` against another member, until the stray `@dataclass` decorators are removed fleet-wide. Membership tests (`x in (A, B)`) are also broken by the same quirk.

## 2026-07-01 — pattern — Idempotency: `effect_guard` (#1084) is execution_id-scoped; user-initiated POSTs use `Idempotency-Key` (#18)
**Context**: Brain Orb Phase 4a KB-write actions (trinity-enterprise#61). The issue said "wire the #1084 effect guard," but `idempotency_service.effect_guard` builds its scope as `effect:{execution_id}` and `resolve_and_validate_execution` fail-opens to no-dedup when there's no execution — brain-orb capture/link are user→backend producer POSTs with **no** `execution_id`, so effect_guard would be a silent no-op (or collide on `effect:None`). The correct primitive is the trigger-boundary `Idempotency-Key` path (`idempotency_service.begin/complete/fail` over `make_agent_scope`, Invariant #18).
**Lesson**: `effect_guard` is for **agent-turn side effects** keyed by the per-fire `execution_id` (messages/voip/share_file). For a **user-initiated** write endpoint that creates no execution, use the `Idempotency-Key` header + `begin/complete/fail`, and **fold the action verb into the key** (`{action}:{client_key}`) so a client that reuses one key across different verbs can't replay the wrong verb's snapshot. Also: `Idempotency-Key` dedups the **HTTP POST**, not the downstream sink write — exactly-once at the sink is still the sink's job; and a fresh-key-per-call client (like the orb) only dedups transparent network retries, so guard double-submits client-side too.

## 2026-07-01 — pitfall — A feature kill-switch is only as airtight as its least-gated entry point
**Context**: Brain Orb Phase 4 (trinity-enterprise#61 arc). `BRAIN_ORB_WRITE_ENABLED` gated `/action`/`/actions`/`/postprocess`, but a later commit added `/refresh` — which drives the same agent `action` hook (exec/reindex) — gated only on the *base* `BRAIN_ORB_ENABLED`, not the write flag (final-review F1). Flipping the write kill-switch off left `/refresh` (and thus the action hook) reachable. Recurs because routes reaching a gated subsystem get added incrementally; each new one must re-assert the gate.
**Lesson**: When adding a route/entry point that reaches a flag-gated subsystem (exec surface, write path, paid feature), gate it on the **same** flag as its siblings — audit *every* path into the subsystem, not just the first. Add a test that toggles the specific kill-switch (not just the base flag) and asserts the new route 404s + never calls downstream. Related idempotency footgun surfaced same review: awaiting `audit.log()` **after** `idempotency.complete()` unguarded — an audit throw 500s a *completed* write, and a fresh-key client retry then double-writes; make post-write audit best-effort.
## 2026-07-01 — pitfall — A CSP fetch/media/object directive that omits `blob:` silently breaks browser features using createObjectURL; and there are TWO CSP copies to keep in sync
**Context**: #1400 (misdiagnosed as #784). The Files-tab preview builds a `blob:` URL (`URL.createObjectURL`) and renders it per type: text/download via `fetch(blobUrl)` (governed by `connect-src`), `<img>` (`img-src`), `<video>/<audio>` (`media-src`), and PDF `<embed>` (`object-src` **and** `frame-src` — Chrome renders the embedded PDF in an internal viewer frame, so the blob is checked against `frame-src` too, not just `object-src`). `blob:` is **not** covered by `'self'` — it must be listed explicitly in the specific directive the load uses. `img-src` listed `blob:` (images previewed) but `connect-src` did not (text failed with "Failed to load text content"), and `media-src`/`object-src`/`frame-src` were unset (fell back to `default-src 'self'` → also blocked; the PDF showed Chrome's "This content is blocked"). The breakage is per-feature and invisible until you exercise that exact path — #784's `Content-Disposition: inline` "fix" never touched the CSP, so preview stayed broken across every release since the CSP landed (#549/#641). Trinity has **two** CSP copies that must move together: `src/frontend/security-headers.conf` (prod nginx) and `src/frontend/vite.config.js` (dev-server middleware). Same class already bit #979 (voice `srcdoc` under prod `script-src 'self'`) and #1116/ask-trinity (`connect-src` external origin).
**Lesson**: When a browser feature loads a `blob:`/`data:` URL, add that scheme to the SPECIFIC directive the load uses (`fetch`→`connect-src`, `<img>`→`img-src`, `<video>/<audio>`→`media-src`, `<object>/<embed>`→`object-src`; a PDF `<embed>` **also** needs `frame-src` because Chrome renders it in an internal viewer frame) — `'self'` does not cover `blob:`/`data:`. Change **both** `security-headers.conf` and `vite.config.js` together (a dev-only CSP hides prod-only breakage and vice-versa). "Works for images but not text" ⇒ suspect a `connect-src` gap first. Guard with a test asserting the scheme in each directive of both files (`tests/unit/test_1400_csp_blob_preview.py`).
