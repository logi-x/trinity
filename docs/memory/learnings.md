# Learnings — durable bug-class ledger

Append-only. One entry per *class* of error/pattern worth knowing before the next
plan or review. `/autoplan` reads this before planning; write for that reader.

## 2026-06-23 — pitfall — Adding a column needs BOTH schema.py AND tables.py; parity test guards only one
**Context**: `agent_ownership.voice_name` (trinity-enterprise#28). `tests/unit/test_schema_parity.py` compares `db/schema.py` (raw DDL) against `db/migrations.py` (the SQLite runner) — it does **not** import `db/tables.py` (the SQLAlchemy Core MetaData that Alembic autogenerates from and that every `db/*.py` accessor binds columns against). So a change that adds a column to `schema.py` + `migrations.py` but forgets `tables.py` keeps the parity test green, yet `select(agent_ownership.c.<col>)` blows up at runtime on the first call.
**Lesson**: A schema change touches **four** files — `db/schema.py`, `db/tables.py`, a SQLite migration in `db/migrations.py`, and an Alembic revision under `db/../migrations/versions/`. Don't trust schema-parity to catch a missing `tables.py` Column. The real guard is a db-accessor unit test that executes a live `select(table.c.<new_col>)` against a migrated DB (db_harness) — it fails loudly if `tables.py` was missed.

## 2026-06-28 — pitfall — `@dataclass` on a fieldless str-Enum breaks member equality
**Context**: `TaskExecutionErrorCode(str, Enum)` is decorated `@dataclass` (`services/task_execution_service.py:75`). `@dataclass` synthesises an `__eq__` that compares the (zero) fields, so for enum members `AUTH == BILLING` — and even `TIMEOUT == AUTH` — all evaluate **True**. `apply_result`'s breaker line `envelope.error_code == TaskExecutionErrorCode.AUTH` therefore fired for *any* non-None code reaching it (504→TIMEOUT pre-existing; 429→BILLING after #1085 populated it), wrongly tripping the AUTH dispatch breaker. Found in the #1085 review; fixed by value-comparing (`error_code.value == "auth"`).
**Lesson**: Never put `@dataclass` on an `Enum` subclass. Compare enum codes by `.value` (or plain `is`/identity), never `==` against another member, until the stray `@dataclass` decorators are removed fleet-wide. Membership tests (`x in (A, B)`) are also broken by the same quirk.

## 2026-07-01 — pattern — Idempotency: `effect_guard` (#1084) is execution_id-scoped; user-initiated POSTs use `Idempotency-Key` (#18)
**Context**: Brain Orb Phase 4a KB-write actions (trinity-enterprise#61). The issue said "wire the #1084 effect guard," but `idempotency_service.effect_guard` builds its scope as `effect:{execution_id}` and `resolve_and_validate_execution` fail-opens to no-dedup when there's no execution — brain-orb capture/link are user→backend producer POSTs with **no** `execution_id`, so effect_guard would be a silent no-op (or collide on `effect:None`). The correct primitive is the trigger-boundary `Idempotency-Key` path (`idempotency_service.begin/complete/fail` over `make_agent_scope`, Invariant #18).
**Lesson**: `effect_guard` is for **agent-turn side effects** keyed by the per-fire `execution_id` (messages/voip/share_file). For a **user-initiated** write endpoint that creates no execution, use the `Idempotency-Key` header + `begin/complete/fail`, and **fold the action verb into the key** (`{action}:{client_key}`) so a client that reuses one key across different verbs can't replay the wrong verb's snapshot. Also: `Idempotency-Key` dedups the **HTTP POST**, not the downstream sink write — exactly-once at the sink is still the sink's job; and a fresh-key-per-call client (like the orb) only dedups transparent network retries, so guard double-submits client-side too.

## 2026-07-01 — pitfall — A feature kill-switch is only as airtight as its least-gated entry point
**Context**: Brain Orb Phase 4 (trinity-enterprise#61 arc). `BRAIN_ORB_WRITE_ENABLED` gated `/action`/`/actions`/`/postprocess`, but a later commit added `/refresh` — which drives the same agent `action` hook (exec/reindex) — gated only on the *base* `BRAIN_ORB_ENABLED`, not the write flag (final-review F1). Flipping the write kill-switch off left `/refresh` (and thus the action hook) reachable. Recurs because routes reaching a gated subsystem get added incrementally; each new one must re-assert the gate.
**Lesson**: When adding a route/entry point that reaches a flag-gated subsystem (exec surface, write path, paid feature), gate it on the **same** flag as its siblings — audit *every* path into the subsystem, not just the first. Add a test that toggles the specific kill-switch (not just the base flag) and asserts the new route 404s + never calls downstream. Related idempotency footgun surfaced same review: awaiting `audit.log()` **after** `idempotency.complete()` unguarded — an audit throw 500s a *completed* write, and a fresh-key client retry then double-writes; make post-write audit best-effort.
## 2026-07-01 — pitfall — A CSP fetch/media/object directive that omits `blob:` silently breaks browser features using createObjectURL; and there are TWO CSP copies to keep in sync
**Context**: #1400 (misdiagnosed as #784). The Files-tab preview builds a `blob:` URL (`URL.createObjectURL`) and renders it per type: text/download via `fetch(blobUrl)` (governed by `connect-src`), `<img>` (`img-src`), `<video>/<audio>` (`media-src`), and PDF `<embed>` (`object-src` **and** `frame-src` — Chrome renders the embedded PDF in an internal viewer frame, so the blob is checked against `frame-src` too, not just `object-src`). `blob:` is **not** covered by `'self'` — it must be listed explicitly in the specific directive the load uses. `img-src` listed `blob:` (images previewed) but `connect-src` did not (text failed with "Failed to load text content"), and `media-src`/`object-src`/`frame-src` were unset (fell back to `default-src 'self'` → also blocked; the PDF showed Chrome's "This content is blocked"). The breakage is per-feature and invisible until you exercise that exact path — #784's `Content-Disposition: inline` "fix" never touched the CSP, so preview stayed broken across every release since the CSP landed (#549/#641). Trinity has **two** CSP copies that must move together: `src/frontend/security-headers.conf` (prod nginx) and `src/frontend/vite.config.js` (dev-server middleware). Same class already bit #979 (voice `srcdoc` under prod `script-src 'self'`) and #1116/ask-trinity (`connect-src` external origin).
**Lesson**: When a browser feature loads a `blob:`/`data:` URL, add that scheme to the SPECIFIC directive the load uses (`fetch`→`connect-src`, `<img>`→`img-src`, `<video>/<audio>`→`media-src`, `<object>/<embed>`→`object-src`; a PDF `<embed>` **also** needs `frame-src` because Chrome renders it in an internal viewer frame) — `'self'` does not cover `blob:`/`data:`. Change **both** `security-headers.conf` and `vite.config.js` together (a dev-only CSP hides prod-only breakage and vice-versa). "Works for images but not text" ⇒ suspect a `connect-src` gap first. Guard with a test asserting the scheme in each directive of both files (`tests/unit/test_1400_csp_blob_preview.py`).

## 2026-07-04 — pattern — An opaque "session identifier" is an overloaded key: split scope / attribution / memory-source before re-keying
**Context**: #903 (Slack thread-scoped sessions). The single `SlackAdapter.get_session_identifier` string secretly encoded **three** orthogonal concerns that a single-user session kept accidentally aligned: conversation *scope* (what replays), speaker *attribution* (who said each turn), and per-user memory *source* (whose turns feed whose MEM-001 profile). The issue's suggested one-liner (drop `sender_id` from the key so channel threads share context) fixed scope but silently broke the other two — `public_chat_messages` had no sender column so multi-participant history replayed as a flat `User:`, and `summarize_user_memory_background` reads *all* session messages and writes them to the *current* user's durable, re-injected memory, so Bob replying in Alice's thread would persist Alice's turns into Bob's profile (cross-user PII leak). Verified in code before planning; scope escalated to add a per-message `sender_email`/`sender_label` so each axis has its own home.
**Lesson**: Before changing what a shared/opaque key encodes, enumerate every concern that key currently disambiguates. A key that "just works" often works because one variable (single participant) collapses several axes into one; loosen that variable (multi-participant thread) and each hidden axis needs its own explicit column/filter. In particular, any "shared context" change to a chat surface must check the **memory summarizer** — it silently reads whole-session history and writes to one user's durable store, so broadening the session scope without a per-sender filter is a cross-user leak.
## 2026-07-04 — pitfall — After the #1093 SQLAlchemy-Core rewrite, `db.get_chat_session()` returns a Pydantic model, not a dict — `.get("k")` raises and is swallowed
**Context**: #1444 review. `db.get_chat_session()` now returns `Optional[ChatSession]` (Pydantic `BaseModel` via `_row_to_chat_session`). The new persistence owner-gate correctly uses attribute access (`session.user_id != user_id`, `chat.py:1016`), but the pre-existing sibling `_finalize_self_task` (`chat.py:1273`) still uses **dict** access `session.get("user_id") == user_id`. `ChatSession` has no `.get()`, so it raises `AttributeError` — caught by the surrounding `except Exception` at :1291 and logged as a warning. Net effect: self-task `inject_result` into a chat session **silently never works**, and its ownership check never actually runs. Pre-#1093 the accessor returned a dict, so `.get()` was correct then; the Core rewrite flipped the return type and this call-site was missed.
**Lesson**: Any accessor rewritten by #1093/#300 from sqlite3.Row/dict to a Pydantic model breaks every `.get("field")` / `row["field"]` caller — and a swallowing `except` turns the break into a silent no-op, not a crash. When touching one owner-check on a `get_chat_session` result, grep **all** call-sites (`grep -rn "get_chat_session\b"`) and confirm each uses attribute access. Add a db-accessor unit test that asserts the returned object is the model and that `.get`/subscript access is not used downstream.
## 2026-07-04 — pitfall — A critical invariant with zero unit exercise lets a `sys.modules` stub leak masquerade as a real violation
**Context**: #1446. Canary B-01 (queue-status coherence, Tier A critical) false-fired under full-suite load — `db.get_queued_count('a1')=1` vs a 0-queued snapshot — but never in isolation. Root cause was **test harness, not production**: several modules do a module-level, never-restored `sys.modules.setdefault("database", MagicMock())` (`test_watchdog_unit.py`, `test_validation.py`, `test_cb_probe_execution_close.py`, …). `setdefault` is first-writer-wins + session-persistent; `int(MagicMock().get_queued_count("a1")) == 1`, so the canary's `_collect_queued_count_via_service` (`from database import db` resolved at *call* time) read `1` for every agent. B-01 had **no end-to-end unit coverage** — only synthetic `_snap`-built comparator tests — so nothing exercised the real `collect_snapshot`→accessor path where the leak bites. On the shipping SQLite backend the two B-01 read paths (raw `sqlite3` `DB_PATH` vs `get_engine()`) resolve to the same file → no production coherence bug; the Postgres divergence (raw-sqlite readers vs `get_engine()`) is a real but deferred gap, tolerable only because the canary is default-OFF.
**Lesson**: (1) A canary/invariant check that only has *synthetic-snapshot* tests is untested where it actually runs — give every invariant one end-to-end test that drives `collect_snapshot()` against the fixture DB so a harness/wiring leak can't hide. (2) Any test reading an ambient `sys.modules["X"]` (esp. `database`) is at the mercy of leaked stubs; the fixture must **own** that dependency via `monkeypatch.setitem` (auto-restored, last-write-wins overrides a leak), and the stub should **delegate to the real backend seam** (`get_engine()` over the temp DB) — not a hand-fed constant, or the coherence test degrades to a tautology. (3) `sys.modules.setdefault(...)` at module scope is the worst offender: first-writer-wins means it survives even an explicit preload and poisons every downstream `from X import ...` for the whole session (the `test_904 database.db` leak family).

## 2026-07-04 — pitfall — `update = none` submodules: skip exits 0, `--init` copies the strategy local, and existing clones inherit the flip asymmetrically
**Context**: #1443 made both private submodules (`.claude`, `src/backend/enterprise`) `update = none` so OSS clones init without credentials. Three git behaviors (all empirically verified) make naive rollouts break silently: (1) a skipped submodule **exits 0** — any script judging init success by exit code (deploy-dev.yml did) takes its success branch while the checkout is absent/stale; (2) **every** init path (plain `--init`, `--init --checkout`, `clone --recurse-submodules`) **copies `update = none` into `.git/config`** — `--init --checkout` is only special in that it additionally clones once — so later plain `git submodule update` silently skips again; a one-shot flag is not a durable opt-in; (3) existing clones are protected from a `.gitmodules` flip **only if** an `update` key existed there at init time (init copied it into local config, which wins): `.claude` clones were protected (`update = checkout` predated the flip), enterprise clones were not.
**Lesson**: When flipping a submodule to `update = none`: patch every automation that inits it to (a) set the durable local override `git config submodule.<path>.update checkout` before init and (b) judge success by a populated marker file, never exit code; document the config-first opt-in (config line, then plain `--init`) as the primary instruction; and check `git config submodule.<path>.update` on existing clones to know who silently changes behavior.

## 2026-07-06 — pitfall — Stripping tzinfo without converting mis-compares non-UTC datetimes; schedules store AWARE next_run_at
**Context**: #1472. `agent_schedules.next_run_at` is written by `_get_next_run_time` as a **timezone-aware** value in the schedule's own zone (e.g. Europe/Kiev `+03:00`). `_get_missed_schedules` (`src/scheduler/service.py`) did `expected.replace(tzinfo=None)` — stripping the offset **without converting** — so a Kiev schedule compared 3h off and a genuinely-missed window looked "future" and was skipped. Separately, a naive `stored > datetime.utcnow()` against an aware value raises `TypeError: can't compare offset-naive and offset-aware` (bit the new advance-guard + a test assert). The scheduler helper, the new advance-guard, and the new canary E-06 check all needed the same UTC-aware handling.
**Lesson**: When comparing a stored schedule/execution timestamp, **convert to UTC, never strip tzinfo**: `dt.astimezone(pytz.utc)` for an aware value, `dt.replace(tzinfo=pytz.utc)` only when it's known-naive-UTC — then compare against `datetime.now(pytz.utc)`. `replace(tzinfo=None)` on an offset-bearing value silently shifts it by the offset; a bare `aware > naive` crashes. Any codepath touching `next_run_at`/`last_run_at`/`started_at` of a non-UTC schedule is in this class.

## 2026-07-06 — pattern — Validate a config with the SAME parser the executor uses, or bad configs pass then fail silently
**Context**: #1472. The schedule create/update API validated cron with a bare `croniter(expr)` and never validated the timezone, but the dedicated scheduler registers via a strict 5-field `_parse_cron` + APScheduler `CronTrigger` + `pytz.timezone`. `@daily`, 6-field seconds-crons, quartz `L`/`#`, and any bad timezone **clear croniter** but throw in `_add_job` — the schedule is accepted, never registers, never fires, and its `next_run_at` freezes ("Next: Nd ago"). The producer-side validator was looser than the consumer-side parser.
**Lesson**: When an API validates input a **different** component will later parse/execute, validate with that component's exact parser, not a convenient looser one. For schedules the parity primitive is `services/schedule_validation.py` (mirrors `src/scheduler/service.py::_parse_cron` + `CronTrigger` + `pytz`) — keep them in sync. Pair it with executor-side resilience (classify a failed registration transient-retry vs permanent-bounded) so a config that still slips through is never a permanent silent orphan.

## 2026-07-06 — pitfall — A new db-layer accessor needs an explicit DatabaseManager facade delegation; wholesale-mocked `database` tests are structurally blind to it
**Context**: trinity-enterprise#93 review (C1). `get_git_config_agent_names_for_repo` was added to `ScheduleOperations` (db/schedules.py) and called via `db.` in crud — but `DatabaseManager` (database.py) has no `__getattr__` fallback; every accessor needs a hand-written one-line delegation. The call would have raised `AttributeError` on the first real request. The 35-test unit suite stayed green because it mocks the entire `database` module with `MagicMock`, which synthesizes any attribute.
**Lesson**: When adding a db-layer method, grep `database.py` for a sibling's delegation and add yours next to it — the facade is manual, not dynamic. And every new accessor needs one test that touches the REAL facade (e.g. `DatabaseManager.__new__` + a mocked `_schedule_ops`, or a db_harness live-select) — a `MagicMock`-ed `database` module can never catch a missing delegation, a typo'd method name, or a wrong return shape.

## 2026-07-07 — pitfall — Any subprocess the agent-server spawns is invisible to the orphan sweep unless explicitly vouched for
**Context**: #1501. `resolve_allowlist`'s hard-protect walk goes from the sweeper UP to PID 1 — never down to the agent-server's other children — so a bare `asyncio.create_subprocess_exec` child of the agent-server (a brain-orb convention hook) matched none of the four allowlist categories and the 30s periodic sweep SIGKILLed it mid-run (the 180s `refresh` reliably → exit -9 → 502). #1494 had already fixed this exact class for post-voice `claude -p` by dispatching a registered execution; `refresh` was left on the bare-hook path. Same investigation surfaced two siblings: gemini's 3 post-turn sweeps ran with NO `extra_pids` (a finishing Gemini turn could kill concurrent in-flight work — the #912 class, unfixed on that path), and user `persistent-processes.allow` patterns protected only the bare matched pid, so an allowlisted daemon's worker children were reaped anyway.
**Lesson**: A new agent-server spawn site (hook, helper subprocess) must either wrap the run in `ProcessRegistry.add_transient_pid`/`remove_transient_pid` (finally-paired, fail-open) or dispatch as a registered execution — otherwise it dies whenever it straddles a sweep tick (P ≈ runtime/30s, ≈certain past 30s). And a new `kill_cgroup_orphans` call site must pass `extra_pids=active_execution_pids()` — a bare call kills concurrent in-flight work. Known residual exposure: `agent_server/routers/git.py` `subprocess.run` children (short-lived git ops; follow-up candidate alongside the descendants_of(sweep_pid) seam discussion). Test-side corollary (the #1446 class, re-confirmed by this PR's first CI run): production code that resolves the registry via a CALL-TIME lazy import cannot be stubbed by patching an attribute on the module object your test file imported — sibling test modules (`test_agent_server_auto_sync.py`, `test_git_status_dual_ahead_behind.py`) evict `agent_server*` from `sys.modules` at module level during collection, so the lazy import re-creates a fresh module your patch never touched (fails in the full suite, passes in isolation). Own the key instead: `monkeypatch.setitem(sys.modules, "agent_server.services.process_registry", stub_module)`.
