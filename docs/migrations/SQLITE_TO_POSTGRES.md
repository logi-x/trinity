# SQLite → PostgreSQL Migration & SQLite End-of-Support

> **SQLite end-of-support date: September 1, 2026.**
> After this date, PostgreSQL is the supported database backend for Trinity. SQLite
> remains the zero-config default **until** then; nothing breaks before the date.
> This is a decision/announcement document (issue #1278) — the SQLite code removal
> is tracked separately under the migration work (#300 / #1183 / #746).

## TL;DR

- Trinity currently runs on **either** SQLite (default) **or** PostgreSQL, on two
  separate migration tracks (bespoke SQLite runner + Alembic for Postgres, #1183).
- The long-term direction is **PostgreSQL-only**. SQLite support is scheduled to be
  retired on **September 1, 2026**.
- If you run the default SQLite setup, **plan to move to PostgreSQL before that date**.
- Switching the backend for a **fresh** deployment is a config change
  (`DATABASE_URL`); migrating an **existing deployment's data** needs a deliberate
  cutover — see [Migrating an existing deployment](#migrating-an-existing-deployment).

## Why this is happening

PostgreSQL support landed via the dual-track migration system (#300 configurable
backend, #1183 Alembic for Postgres). Maintaining two database backends — and
authoring **two** migrations for every schema change — is a long-term tax. Once
the single-source-of-truth schema consolidation lands (#746, `db/tables.py`
MetaData → autogenerated revisions), the SQLite bespoke runner is retired and
PostgreSQL becomes the one supported backend.

## What changes, and when

| | Before Sept 1, 2026 | After Sept 1, 2026 |
|---|---|---|
| SQLite (default) | ✅ Fully supported | ⛔ End-of-support (no further fixes/migrations) |
| PostgreSQL | ✅ Supported (opt-in via `DATABASE_URL`) | ✅ Supported backend |
| Action for operators | None required; migrate at your convenience | Run on PostgreSQL, or pin a pre-EOL release |

Nothing in your running instance changes on the date itself — SQLite installs keep
working — but they stop receiving schema migrations and fixes, so staying on SQLite
past EOL means pinning an older Trinity release.

## Switching to PostgreSQL (fresh deployment)

For a new instance, PostgreSQL is a configuration change. The backend builds the
full schema on first boot via Alembic (`alembic upgrade head`).

1. Bring up the bundled Postgres service:
   ```bash
   docker compose --profile postgres up -d postgres
   ```
2. Set `DATABASE_URL` in `.env` (see `.env.example`):
   ```bash
   DATABASE_URL=postgresql://trinity:your-postgres-password@postgres:5432/trinity
   ```
   Leave `DATABASE_URL` unset to keep the SQLite default.
3. Restart the backend (and scheduler). On startup the non-SQLite path runs
   `db/alembic_runner.upgrade_to_head()`, which builds the schema from the
   `0001_baseline` revision (the same DDL as `init_schema_postgres`).
4. Verify (see [Verifying the cutover](#verifying-the-cutover)).

## Migrating an existing deployment

The dual-track system migrates **schema**, not your existing **data** — there is no
turnkey SQLite→Postgres data-copy tool today (tracked with the migration work in
#300). Plan a deliberate cutover:

1. **Back up first**, always:
   ```bash
   scripts/deploy/backup-database.sh
   ```
2. Stand up PostgreSQL (`docker compose --profile postgres up -d postgres`).
3. Move your data using one of:
   - **Fresh start (simplest):** point `DATABASE_URL` at Postgres and let Trinity
     build a clean schema; recreate agents/config there. Best when the SQLite
     history isn't worth carrying forward.
   - **Data copy (manual):** export the SQLite tables and load them into the
     matching Postgres tables (e.g. via `pgloader` or a per-table dump/restore),
     respecting the schema that `alembic upgrade head` created. Validate row counts
     against the SQLite source before cutting over.
4. Set `DATABASE_URL`, restart the backend + scheduler, and verify.

> Keep the SQLite `trinity.db` backup until you've confirmed the Postgres instance
> is correct — the cutover is one-way in practice.

## Verifying the cutover

- `GET /health` returns healthy (and reports no pending migration).
- Alembic is at head:
  ```bash
  docker exec trinity-postgres psql -U trinity -d trinity -tAc \
    "SELECT version_num FROM alembic_version;"
  ```
- Agents list, schedules, and recent executions render as expected in the UI.

## Release-notes announcement (draft copy for the next release)

> **PostgreSQL is now the forward path; SQLite support ends September 1, 2026.**
> Trinity has supported PostgreSQL alongside SQLite since the dual-track database
> work (#300, #1183). We're committing to a single supported backend: **SQLite
> reaches end-of-support on September 1, 2026.** SQLite remains the zero-config
> default until then and existing installs keep working, but after the date SQLite
> stops receiving schema migrations and fixes. Operators on SQLite should plan to
> move to PostgreSQL ahead of the cutoff — see
> [SQLite → PostgreSQL Migration](../migrations/SQLITE_TO_POSTGRES.md).
