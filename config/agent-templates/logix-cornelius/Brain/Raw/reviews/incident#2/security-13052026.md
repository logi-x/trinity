```bash
cd /home/logix/experts

```

You are the main security incident orchestrator for the Experts project.

I will provide a list of security incidents. Run the full process end-to-end in this order:

1. Investigate
   Spawn/read from:
   - security-auditor
   - dependency-auditor if dependencies/CVEs are involved
   - architecture-reviewer
   - qa-tester

2. Triage
   For each incident classify:
   - severity
   - exploitability
   - reachability
   - affected files/systems
   - whether it is hotfix, normal fix, or false positive

3. Plan
   Ask the relevant specialist to propose the safest remediation:
   - frontend-developer for frontend/API route fixes
   - db-engineer for Prisma/Postgres/migration issues
   - payments-specialist for ZATCA/payments/ledger issues
   - realtime-engineer for WebSocket/presence issues
   - devops-engineer/platform-operator for infra/config issues

4. Implement
   Use exactly ONE writing agent.
   Do not allow multiple agents to edit files.
   Work only in the current directory.
   Do not create worktrees.
   Do not switch branches.
   Do not push until review passes.

5. Review
   Spawn read-only review from:
   - security-auditor
   - code-reviewer
   - qa-tester
   - architecture-reviewer if architectural changes were made
   - release-manager if migrations/config/deploy changes were made

6. Fix review findings
   Apply only necessary changes.
   Avoid broad refactors or unrelated cleanup.

7. Validate
   Run relevant tests/typechecks/lints for the changed scope.
   If a command fails, explain whether it is caused by this change or existing project state.

8. Commit and push
   Only after review and validation.
   Make clean, logical commits.
   Push the current branch only.
   Never force-push.

Before doing anything, print:

- pwd
- git branch --show-current
- git status -sb

- Treat this as a security remediation workflow.
- Prefer minimal surgical fixes.
- Verify reachability; do not fix purely theoretical issues unless they are clearly risky.
- Do not create new branches or worktrees.
- Do not modify unrelated files.
- Do not hide uncertainty.
- At the end, provide:
  - incidents fixed

  - files changed

  - commits created
  - push result
  - remaining risks

## Security Review Incident #2

### Report #1

Scheduled application security review result

Found 1 validated medium issue. No validated high/critical issues found.

Medium: authenticated arbitrary public upload with executable content
• Location: apps/experts-app/app/api/v1/internal/upload/route.ts, apps/experts-app/src/lib/storage/commands/upload-public-asset.command.ts
• Attacker: any authenticated user, including a regular/student account.
• Controlled input: multipart file, declared MIME type, domain, and entityId.
• Path: POST /api/v1/internal/upload only checks auth(), does not verify role or ownership of the supplied upload context, accepts attacker-chosen domain/entityId, allows text/html and text/javascript for type=file, then writes to public R2 with that ContentType.
• Impact: public trusted-asset-host file/HTML/JS hosting under arbitrary entity namespaces, enabling storage/bandwidth abuse and phishing/malware or script execution on the asset origin.
• Highest-leverage remediation: add a strict domain allowlist plus per-domain object ownership/creator authorization before upload; block executable web MIME types or force download-safe headers (application/octet-stream, Content-Disposition: attachment, X-Content-Type-Options: nosniff); add quota/rate limits.

Downgraded candidates were not reported: Tabby webhook signature skipping is limited to non-production mode and checked deploy compose defaults NODE_ENV=production; the broad next/image remote patterns did not validate as private-network SSRF because current Next defaults block local IP optimization unless explicitly enabled.
