# Security Incident Remediation Workflow

```bash
cd /home/logix/experts
git checkout -b fix/security-incidents-YYYYMMDD
```

You are the main security incident orchestrator for the Experts project.

I will provide a list of security incidents. Run the full process end-to-end in this order:

1. Investigate
   Spawn/read from:
   - security-auditor
   - dependency-auditor if dependencies/CVEs are involved
   - architecture-reviewer
   - qa-tester

2. Triage
   For each incident classify:
   - severity
   - exploitability
   - reachability
   - affected files/systems
   - whether it is hotfix, normal fix, or false positive

3. Plan
   Ask the relevant specialist to propose the safest remediation:
   - frontend-developer for frontend/API route fixes
   - db-engineer for Prisma/Postgres/migration issues
   - payments-specialist for ZATCA/payments/ledger issues
   - realtime-engineer for WebSocket/presence issues
   - devops-engineer/platform-operator for infra/config issues

4. Implement
   Use exactly ONE writing agent.
   Do not allow multiple agents to edit files.
   Work only in the current directory.
   Do not create worktrees.
   Do not switch branches.
   Do not push until review passes.

5. Review
   Spawn read-only review from:
   - security-auditor
   - code-reviewer
   - qa-tester
   - architecture-reviewer if architectural changes were made
   - release-manager if migrations/config/deploy changes were made

6. Fix review findings
   Apply only necessary changes.
   Avoid broad refactors or unrelated cleanup.

7. Validate
   Run relevant tests/typechecks/lints for the changed scope.
   If a command fails, explain whether it is caused by this change or existing project state.

8. Commit and push
   Only after review and validation.
   Make clean, logical commits.
   Push the current branch only.
   Never force-push.

Before doing anything, print:

- pwd
- git branch --show-current
- git status -sb

Rules:

- Treat this as a security remediation workflow.
- Prefer minimal surgical fixes.
- Verify reachability; do not fix purely theoretical issues unless they are clearly risky.
- Do not create new branches or worktrees.
- Do not modify unrelated files.
- Do not hide uncertainty.
- At the end, provide:
  - incidents fixed
  - incidents rejected/false-positive
  - files changed
  - tests run
  - commits created
  - push result
  - remaining risks

Commit policy:

- No agent may commit automatically.
- Implementation completion means code changes are present in the working tree, reviewed, and validated.
- Only the main session may ask for a commit.
- The exact phrase required for committing is: "commit now".
- The exact phrase required for pushing is: "push now".
- If any sub-agent attempts to commit without that phrase, stop and report a policy violation.

## Security Review Incident #1

### Report #1

Scheduled application-security review found validated medium+ issues:

Critical: unauthenticated account/credential disclosure
Location: apps/experts-app/app/api/v1/internal/debug/accounts/route.ts
Attack path: anonymous attacker controls ?id=<userUuid> on GET /api/v1/internal/debug/accounts; /api bypasses proxy auth in apps/experts-app/proxy.ts.
Impact: returns full User plus Account rows containing passwordHash, accessToken, and refreshToken per apps/experts-prisma/prisma/schema.prisma.
Remediation: remove production debug routes or require admin/secret auth; return explicit safe select fields only.

Critical: unauthenticated community post update IDOR
Location: apps/experts-app/app/api/v1/community/posts/[id]/route.ts
Attack path: anonymous attacker controls post UUID and PUT JSON body; handler updates prisma.post.update({where:{id}}) with no auth() or ownership check.
Impact: arbitrary post defacement/hiding/feed manipulation, including content, isPublished, and isPinned.
Remediation: require session, validate UUID/body with Zod, enforce owner/admin authorization, and restrict admin-only fields.

High: unauthenticated internal debug integration abuse
Locations: apps/experts-app/app/api/v1/internal/debug/redis/route.ts, .../notifications/route.ts, .../mailtrap/route.ts, .../env/route.ts
Attack path: anonymous attacker reaches /api/v1/internal/debug/\*; handlers have no auth.
Impact: leaks REDIS_URL/mail config, writes/publishes to Redis test channels/caches, and can trigger email/Slack notification delivery with attacker-controlled payloads.
Remediation: delete or compile out debug routes in production; otherwise gate with admin plus service-token auth and never echo secrets.

High: stored XSS through markdown rendering
Location: apps/experts-app/src/components/markdown/MarkdownPreview.tsx
Attack path: authenticated content author/commenter controls stored markdown (POST /api/v1/community/posts, comments, profile/course content); renderer uses skipHtml={false}, rehypeSanitize is disabled, and allowElement only checks tag-name shape.
Impact: raw HTML such as dangerous embeds/attributes can execute in viewers’ browsers, enabling actions as the victim session.
Remediation: set skipHtml to true or enable rehype-sanitize with a strict schema; disallow script, iframe, event attributes, srcdoc, and dangerous URL protocols.

### Report #2

Scheduled app-security review: validated medium+ findings

1. Critical: unauthenticated credential/token disclosure
   GET /api/v1/internal/debug/accounts?id=<uuid> in apps/experts-app/app/api/v1/internal/debug/accounts/route.ts has no auth. An anonymous attacker controls id (user UUIDs are exposed by public profile APIs) and receives the user plus linked Account rows, including passwordHash, accessToken, and refreshToken. Impact: account takeover/offline cracking/OAuth token abuse. Remediation: remove this debug route from production or require admin auth, and never return account secrets.

2. Critical: Tabby webhook forgery when TABBY_WEBHOOK_SECRET is unset
   POST /api/webhooks/tabby flows through TabbyWebhookProvider.verify, which returns without verifying a signature if the secret is absent. An attacker controlling webhook JSON (id, status, order.reference_id) can complete/cancel matching Tabby enrollments or event registrations in tabby.webhook.handler.ts. Impact: unpaid access or payment-state corruption. Remediation: make TABBY_WEBHOOK_SECRET mandatory in production, reject unsigned webhooks, and verify provider state before mutating local state.

3. High: unauthenticated internal debug utility abuse
   /api/v1/internal/debug/mailtrap, /notifications, /redis, and /env are not auth-gated. Anonymous attackers control host/token/to, notification key/payload/channels, and Redis debug actions. Impact: SSRF from the Mailtrap route, arbitrary notification/email/Slack sends, Redis URL disclosure and pub/sub/cache abuse, mail config disclosure, plus secret logging. Remediation: delete these routes from production or gate all internal/debug routes behind admin auth plus IP allowlisting; remove secret logging.

4. High: stored XSS through UGC Markdown rendering
   Authenticated users control community post/comment content; pages render it via MarkdownPreview. @uiw/react-markdown-preview enables rehype-raw when skipHtml={false}, while rehypeSanitize is disabled and URL transformation is effectively identity. Impact: stored same-origin script execution for viewers of malicious posts/comments. Remediation: disable raw HTML for UGC (skipHtml), enable rehype-sanitize with a narrow schema, and enforce safe URL protocols.

### Report #3

Scheduled app-security review found validated medium+ issues in apps/experts-app. /api paths are explicitly passed through by proxy.ts, so these route handlers are reachable unless blocked outside the app.

Critical: unauthenticated account credential dump
Location: app/api/v1/internal/debug/accounts/route.ts
Attacker/input: anonymous client controls id; public profiles expose user UUIDs.
Path/impact: GET /api/v1/internal/debug/accounts?id=<userId> returns raw User plus Account rows, including passwordHash, OAuth accessToken/refreshToken, email, and admin flags.
Remediation: remove this debug route from production, or require admin auth and return an explicit safe select only. Never serialize Account secrets.

High: unauthenticated SSRF and secret logging via Mailtrap debug
Location: app/api/v1/internal/debug/mailtrap/route.ts
Attacker/input: anonymous client controls host, token, to, and subject.
Path/impact: GET /api/v1/internal/debug/mailtrap?host=... calls fetch(${host}/api/send`)`; it also logs/observes MAILTRAP_API_KEY, sender config, recipient, and HTML. This enables server-side requests to attacker-chosen/internal hosts and can push secrets/PII into logs/diagnostics.
Remediation: delete or production-disable the route; never accept request-supplied mail host/token; redact all secrets from logs.

High: unauthenticated diagnostics log exposure
Location: app/api/v1/console/diagnostics/history/route.ts and app/api/v1/console/diagnostics/stream/route.ts
Attacker/input: anonymous client sends GET, no auth required.
Path/impact: history returns the last 100 systemEvent records including JSON context; stream subscribes to diagnostic events. Observability persists event context via DatabaseObserver, so secrets/PII emitted by payment, email, and debug paths can be exposed.
Remediation: require admin authorization on both routes and redact/suppress sensitive fields before persistence or streaming.

High: unauthenticated notification/email/Slack fan-out
Location: app/api/v1/internal/debug/notifications/route.ts
Attacker/input: anonymous client controls key, channels, and POST payload.email/template fields.
Path/impact: route calls notify(...); notify sends email to payload.email and can send Slack/Telegram/webhook messages. Attackers can generate platform-branded phishing/spam using real notification templates.
Remediation: remove from production or gate with admin auth, fixed test recipients, rate limits, and strict payload validation.Medium/High: unauthenticated Redis debug control and URL disclosure
Location: app/api/v1/internal/debug/redis/route.ts
Attacker/input: anonymous client controls POST action and postId.
Path/impact: route performs Redis writes/deletes/publishes and returns process.env.REDIS_URL when set, which commonly contains credentials. This enables cache/pub-sub abuse and infrastructure disclosure.
Remediation: remove from production or require admin/cron auth; do not return connection URLs; isolate diagnostics from mutable cache operations.

### Report #4

Scheduled application-security review found validated medium+ issues:

Critical — Unauthenticated account secret disclosure in apps/experts-app/app/api/v1/internal/debug/accounts/route.ts
Attacker: anyone who can reach the app. Input: id query param. Path: /api/v1/internal/debug/accounts?id=<public user UUID> has no auth()/admin gate and returns prisma.account.findMany() rows. Impact: exposes passwordHash, OAuth accessToken, and refreshToken fields for the target user. Remediation: remove debug route from production; if retained, require admin + IP/secret gate and use select to exclude all credential/token fields.

High — Unauthenticated debug operations in apps/experts-app/app/api/v1/internal/debug/{mailtrap,notifications,redis,env}/route.ts
Attacker: unauthenticated client. Input: query/body controls notification keys, channels, Mailtrap host/token, and Redis test actions. Path: /api is skipped by proxy.ts, and these routes have no route-level auth. Impact: server-side outbound requests, notification/email/Slack abuse, Redis probe/write/publish, and mail config disclosure/logging. Remediation: delete/compile out debug routes in production or require admin + static ops secret + network allowlist; never log mail secrets.

High — Client-controlled session update can grant roles in apps/experts-app/src/lib/auth.ts
Attacker: any logged-in user. Input: Auth.js session update payload containing session.user.roles. Path: JWT callback writes token.roles = session.user.roles, then many /api/v1 routes trust session.user.roles?.includes("admin"); e.g. course approval in app/api/v1/courses/[id]/approve/route.ts and src/lib/courses/catalog/guards/course-access.guard.ts short-circuit admin access without DB verification. Impact: privilege escalation to admin/instructor actions and cross-course writes. Remediation: never accept roles from session update; rehydrate roles/admin flags from DB in JWT/session callbacks and use requireAdmin()/DB-backed permission checks in routes.

High, config-dependent — Tabby webhook fails open when TABBY_WEBHOOK_SECRET is absent in apps/experts-app/src/notifications/channels/webhook/providers/tabby.provider.ts
Attacker: anyone who can POST to /api/webhooks/tabby. Input: forged Tabby JSON. Path: verify() returns without signature validation when webhookSecret is missing, including production; handler then calls enrollment/registration completion or cancellation logic in tabby.webhook.handler.ts. Impact: forged payment success/failure events can complete or cancel enrollments/registrations for known provider refs. Remediation: fail closed in production when the secret is absent; require signature verification for all production Tabby webhooks and alert on missing config.
