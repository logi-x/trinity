## Security Review Incident #8

### Report

Scheduled application-security review: validated findings

1. High: Public course detail leaks paid/draft lesson content
   • Location: app/api/v1/courses/[id]/route.ts -> src/lib/courses/catalog/handlers/course-detail.handler.ts -> src/lib/courses/catalog/mappers/course.mapper.ts
   • Attacker/input: unauthenticated user controls course UUID in GET /api/v1/courses/{id}.
   • Path: route allows optional session, handler loads the course by id with modules/lessons and no publishingStatus or enrollment/owner check, mapper returns each lesson videoUrl and content.
   • Impact: exfiltration of paid lesson media/content and unpublished course material when the course UUID is known/leaked.
   • Remediation: split public vs enrolled/owner course projections; require published status for anonymous reads and only return lesson content/videoUrl for free-preview lessons, enrolled users, owners, or admins.

2. Medium: Unauthenticated arbitrary-user presence lookup
   • Location: app/api/v1/internal/presence/[userId]/route.ts, app/api/v1/internal/presence/batch/route.ts
   • Attacker/input: unauthenticated user controls userId path param or up to 100 comma-separated userIds.
   • Path: GET handlers do not call auth() and return Redis presence/last-seen fields directly.
   • Impact: remote user monitoring/correlation via isOnline, lastSeenAt, and lastActivityAt for arbitrary user IDs.
   • Remediation: require auth and restrict to self/admin/approved viewer relationships, or reduce public output to coarse online indicators only where explicitly intended.

3. Medium: Unauthenticated viewer snapshots expose behavioral/profile metadata
   • Location: app/api/v1/internal/viewers/route.ts
   • Attacker/input: unauthenticated user controls type (post, course, event) and resource id in GET /api/v1/internal/viewers.
   • Path: route validates only type/id, reads Redis viewer IDs, and returns viewer count plus up to 10 user identities/profile fields.
   • Impact: disclosure of who is viewing a given resource, enabling behavioral tracking for public/guessable content IDs.
   • Remediation: require authorization aligned with the resource visibility, return aggregate counts to anonymous callers, and only expose viewer identities to authorized participants/owners.

Reviewed auth/authz routes, internal realtime/presence surfaces, webhooks/uploads, raw SQL, shell execution, redirects, and templating sinks. No PR opened.
