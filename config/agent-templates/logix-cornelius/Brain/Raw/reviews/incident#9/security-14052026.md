## Security Review Incident #9

### Report

Carried forward from incident #6 review. Found while applying the buffer-size recheck across the three upload routes — the same defensive scope review surfaced this attacker-controlled-extension gap in both thumbnail routes.

Medium: thumbnail routes derive R2 object extension from the client-supplied filename, not from the validated MIME type

- **Locations:**
  - `apps/experts-app/app/api/v1/courses/[id]/thumbnail/route.ts:61`
  - `apps/experts-app/app/api/v1/events/[id]/thumbnail/route.ts:61`
- **Offending lines:**
  ```ts
  const extension = file.name.split(".").pop()?.toLowerCase() || "jpg";
  const key = `assets/courses/${courseId}/${fileId}.${extension}`;
  ```
- **Attacker:** any authenticated owner of a course or event host (post-#3 fix — these routes are now ownership-gated).
- **Path:**
  1. Attacker uploads a real PNG (`Content-Type: image/png`) — MIME allowlist check passes.
  2. Attacker names the file `evil.js` (or `evil.html`, `evil.svg`, `payload.xss`).
  3. The route accepts `file.name.split(".").pop()` as the truth, so the object lands at `assets/courses/<courseId>/<fileId>.js`.
  4. The CDN, depending on its content-type-from-extension logic, may serve the object with `Content-Type: application/javascript` (or `text/html`) regardless of the bytes inside. Even with `nosniff` (incident #7), the **declared** content-type now matches an executable type.
  5. If the app's CSP ever allowlists `cdn.experts.com.sa` as a `script-src` (or doesn't ship a CSP at all — current state per incident #7 investigation), the attacker has uploaded an executable asset under the platform's own CDN origin.
- **Reachability assessment:**
  - **Owner-gated** post-#3, so this requires authenticated owner abuse — but owner accounts are not trust-equivalent to platform admins, and free-tier course creation is open.
  - **Real risk surface** if any of these is true (each independently): CSP is permissive on `cdn.experts.com.sa`, the Cloudflare worker fronting R2 derives content-type from URL extension, or any code path embeds the CDN URL into a `<script src>` / `<iframe>` / direct user-redirect.
  - Cross-cuts incident #7 (CDN hardening): the two defenses reinforce each other. Fixing only the CDN side is insufficient if the path component is still attacker-controlled.
- **Contrast with `internal/upload/route.ts`:** that route already does the right thing — it has a `MIME_TO_EXT` lookup table and uses `MIME_TO_EXT[normalizedMimeType] || "bin"` to derive the extension from the validated MIME, ignoring `file.name` entirely. The thumbnail routes diverged at some point and were never realigned.
- **Highest-leverage remediation:**
  1. Add (or share) a `MIME_TO_EXT` mapping covering only the thumbnail-allowed image MIMEs:
     ```ts
     const MIME_TO_EXT: Record<string, string> = {
       "image/jpeg": "jpg",
       "image/jpg": "jpg",
       "image/png": "png",
       "image/webp": "webp",
     };
     const extension = MIME_TO_EXT[file.type] ?? "bin";
     ```
  2. Apply to both `courses/[id]/thumbnail/route.ts` and `events/[id]/thumbnail/route.ts`.
  3. Optional consolidation: extract the table to `src/lib/upload/mime-to-extension.ts` shared by all three upload routes, so `internal/upload/route.ts` no longer redefines the same data. **Out of scope for the security fix** — keep this incident surgical and file the dedup as a refactor follow-up.
  4. Add one test per route: upload an `image/png` with filename `evil.js`, assert the constructed object key ends in `.png`, not `.js`.
- **Affected paths:** only the two thumbnail routes; `internal/upload/route.ts` already does this correctly.

This is a small, surgical fix. Single writing agent, two file edits + two test additions. No DB migration, no infra change.
