## Security Review Incident #4

### Report

Carried forward from incident #2 (PR #308). The authorization hole on `/api/v1/internal/upload` is now closed, but the endpoint still has no per-user rate limiting or storage quota.

Medium: no per-user upload rate limit or storage quota

- **Location:** `apps/experts-app/app/api/v1/internal/upload/route.ts`
- **Attacker:** any authenticated creator/instructor (post-fix the endpoint is gated to entity owners, so the abuser must be a legitimate creator — this is a lower bar than incident #2 but still real).
- **Path:** the only size guard is the per-request `MAX_FILE_SIZE = 50 * 1024 * 1024`. There is no per-user request rate limit, no per-user total storage cap, and no aggregate read of `File` rows to enforce one.
- **Impact:** cost abuse (R2 storage + CDN egress scale linearly with requests); a compromised or malicious instructor can loop 50 MB uploads with no server-side stop. Bandwidth bill, possible CDN performance degradation, junk data in the `File`/`Attachment` tables.
- **Highest-leverage remediation:**
  - Sliding-window per-user rate limit at middleware layer (Upstash/Redis or in-process if no Redis dependency). Suggested: 30 uploads / 5 min per user, 200 uploads / 24 h per user. Configurable via env var.
  - Per-user storage quota check: `prisma.file.aggregate({ where: { /* attachments owned by user */ }, _sum: { size: true } })` before each upload, with a configurable cap (e.g., 5 GB total per non-elite-tier user). Skip the aggregate when under, say, 80% of cap to avoid the query on every request.
  - Decide on a clear UX for the 429/quota-exceeded response (Arabic + English copy in the response body).

This is a defense-in-depth + cost-control fix, not a pure security fix — but it's the natural follow-up to the incident #2 lockdown and should be tracked under the same workstream.
