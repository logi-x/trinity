## Security Review Incident #3

### Report

Discovered while remediating incident #2 (`/api/v1/internal/upload` authorization bypass — PR #308). Same vulnerability class, different routes — explicitly scoped out of incident #2 because the workflow says "do not modify unrelated files."

High: missing ownership check on entity thumbnail routes

- **Locations:**
    - `apps/experts-app/app/api/v1/courses/[id]/thumbnail/route.ts`
    - `apps/experts-app/app/api/v1/events/[id]/thumbnail/route.ts`
- **Attacker:** any authenticated user.
- **Path:** both routes verify session but do NOT verify that `session.user.id` owns the entity (course / event). They then call `uploadPublicAsset` to write a thumbnail to public R2 under the entity's namespace.
- **Reference (correct pattern already in repo):** `apps/experts-app/app/api/v1/community/posts/[id]/thumbnail/route.ts` performs the ownership check via `post.userId !== session.user.id → 403` before writing.
- **Impact:** any authenticated user can overwrite any course's or event's thumbnail image. Reputational/brand damage, defacement, phishing via a recognizable course's thumbnail being swapped for attacker-controlled content.
- **Highest-leverage remediation:** mirror the `community/posts/[id]/thumbnail` pattern — fetch the entity by id, verify the caller is the creator/instructor (for `Course`, check `course.instructors` includes the caller; for `Event`, check the event owner field), 403 otherwise.

Reachability is confirmed by the architecture-reviewer in the incident #2 investigation (see PR #308 conversation / commit f16d8e66 review trail).
