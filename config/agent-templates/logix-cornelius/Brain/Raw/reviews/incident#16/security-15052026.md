# Security Incident Remediation Workflow

```bash
cd /home/logix/experts
git checkout -b fix/security-incidents-YYYYMMDD
```

You are the main security incident orchestrator for the Experts project.

I will provide a list of security incidents. Run the full process end-to-end in this order:

1. Investigate
   Spawn/read from:
   - security-auditor
   - dependency-auditor if dependencies/CVEs are involved
   - architecture-reviewer
   - qa-tester

2. Triage
   For each incident classify:
   - severity
   - exploitability
   - reachability
   - affected files/systems
   - whether it is hotfix, normal fix, or false positive

3. Plan
   Ask the relevant specialist to propose the safest remediation:
   - frontend-developer for frontend/API route fixes
   - db-engineer for Prisma/Postgres/migration issues
   - payments-specialist for ZATCA/payments/ledger issues
   - realtime-engineer for WebSocket/presence issues
   - devops-engineer/platform-operator for infra/config issues

4. Implement
   Use exactly ONE writing agent.
   Do not allow multiple agents to edit files.
   Work only in the current directory.
   Do not create worktrees.
   Do not switch branches.
   Do not push until review passes.

5. Review
   Spawn read-only review from:
   - security-auditor
   - code-reviewer
   - qa-tester
   - architecture-reviewer if architectural changes were made
   - release-manager if migrations/config/deploy changes were made

6. Fix review findings
   Apply only necessary changes.
   Avoid broad refactors or unrelated cleanup.

7. Validate
   Run relevant tests/typechecks/lints for the changed scope.
   If a command fails, explain whether it is caused by this change or existing project state.

8. Commit and push
   Only after review and validation.
   Make clean, logical commits.
   Push the current branch only.
   Never force-push.

Before doing anything, print:

- pwd
- git branch --show-current
- git status -sb

- Treat this as a security remediation workflow.
- Prefer minimal surgical fixes.
- Verify reachability; do not fix purely theoretical issues unless they are clearly risky.
- Do not create new branches or worktrees.
- Do not modify unrelated files.
- Do not hide uncertainty.
- At the end, provide:
    - incidents fixed

    - files changed

    - commits created
    - push result
    - remaining risks

- The exact phrase required for committing is: "commit now".

## Security Review Incident #8

### Report

Scheduled application-security review found validated medium+ issues:

High: paid course completion/certificate can be forged without confirmed payment
Location: app/api/v1/courses/[id]/progress/route.ts, app/api/v1/courses/[id]/modules/[moduleId]/lessons/[lessonId]/watch-progress/route.ts, src/lib/courses/learn/helpers/course-progress.ts, app/api/v1/user/certificates/route.ts, src/lib/courses/enrollments/handlers/course-enroll.handler.ts.
Attack path: an authenticated learner starts paid checkout, which creates a pending enrollment before payment. They can then post forged watch segments/progress for course lessons; those endpoints do not require a completed enrollment. updateCourseEnrollmentProgress updates any enrollment for that user/course and sets completedAt at 100%, while certificate issuance only checks completedAt, not enrollment status or payment confirmation.
Impact: unpaid users can falsify course completion and mint certificates for supported course structures; the same missing guard weakens assessment-gated courses.
Remediation: require completed/paid enrollment or privileged instructor/admin access in all learner progress/watch/assessment mutations, and require enrollment.status === "completed" plus server-derived completion before issuing certificates.

High: paid quiz content and attempts are reachable without course access
Location: app/api/v1/quizzes/[id]/start/route.ts, src/lib/courses/quizzes/handlers/start-quiz.handler.ts, src/lib/courses/catalog/mappers/course.mapper.ts, app/api/v1/courses/[id]/modules/route.ts.
Attack path: published course/module responses expose quiz IDs even for non-enrolled users. Any authenticated user can call POST /api/v1/quizzes/[id]/start; startQuiz only checks isPublished, then buildAttemptResponse returns question text, options, and assets.
Impact: paid-course quiz content can be accessed and attempts created without buying/enrolling in the course.
Remediation: centralize a course access guard and enforce it before quiz start/resume/submit responses; do not treat isPublished as sufficient for paid course assessments.

High: Noon API credentials are logged during checkout creation
Location: src/lib/payments/gateways/noon/noon.client.ts via POST /api/v1/courses/[id]/enroll with provider noon -> createNoonPaymentIntent -> noonPost.
Attack path: any authenticated user who starts a Noon checkout causes noonPost to execute console.log(makeNoonAuthHeader()), writing the merchant Authorization header derived from NOON_BUSINESS_ID, NOON_APP_ID, and NOON_APP_KEY to logs.
Impact: anyone with access to app logs/observability can recover live Noon merchant API credentials.
Remediation: remove the log statement, rotate Noon credentials, and audit retained logs for prior leakage.
