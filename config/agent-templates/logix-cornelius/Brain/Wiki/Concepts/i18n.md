# i18n — Internationalisation

Multilingual support across the Experts platform (web + iOS).

## Locales

| Code | Language | Direction |
| ---- | -------- | --------- |
| `en` | English  | LTR       |
| `ar` | Arabic   | **RTL**   |
| `es` | Spanish  | LTR       |

**Currency:** Always **SAR** in product UI regardless of locale.

## Architecture — URL-based routing

The app migrated from cookie-based to **URL-based** locale routing (V4). Every user-facing route is prefixed with the locale:

```
/en/courses     ← English
/ar/courses     ← Arabic (RTL)
/es/courses     ← Spanish
/@ahmed         ← Profile routes are NOT locale-scoped (global)
/api/v1/...     ← API routes are NOT locale-scoped
```

### Why URL-based (not cookie-based)

- Each language is independently indexable by search engines
- Clean per-locale sitemaps (`sitemap-en.xml`, `sitemap-ar.xml`)
- Proper `hreflang` alternates
- Shareable URLs carry the correct language

Cookie still exists — used only as UX preference (remembers last choice). It does not control routing.

## App Directory Structure (`apps/experts-app/app/`)

```
app/
├── (i18n)/
│   ├── en/          ← Thin locale wrappers (re-export _shared)
│   ├── ar/
│   ├── es/
│   └── _shared/     ← Actual page implementations live here
├── @[username]/     ← Global profile routes (no locale)
├── api/             ← API routes (no locale)
└── page.tsx         ← Root redirect → /ar or /en based on cookie/default
```

### Locale wrapper pattern

Locale route files are thin re-exports — no duplication:

```typescript
// app/(i18n)/en/courses/page.tsx
import SharedPage from "@/app/(i18n)/_shared/courses/page";
export default async function LocalePage() { return <SharedPage />; }
```

All implementation lives in `_shared/`.

## next-intl Wiring

| File                          | Role                                                            |
| ----------------------------- | --------------------------------------------------------------- |
| `src/i18n/routing.ts`         | Centralized routing config — `Link`, `useRouter`, `usePathname` |
| `src/i18n/request.ts`         | Server-side locale from URL                                     |
| `src/i18n/locale.ts`          | Cookie helpers (UX preference only)                             |
| `src/i18n/messages/{locale}/` | Translation strings, split by domain                            |

**Always import from `@/i18n/routing`**, not from `next/navigation`:

```typescript
// Wrong
import Link from "next/link";
import { useRouter } from "next/navigation";

// Correct
import { Link, useRouter, usePathname } from "@/i18n/routing";
```

## Translation Usage

```typescript
// Server Component / API
const t = await getTranslations("courses.metadata");

// Client Component
const t = useTranslations("courses");
```

Message files in `src/i18n/messages/{locale}/{domain}/`. Same keys across all locales — English is the source of truth.

## Adding a New Page

1. Add message file: `src/i18n/messages/{en,ar,es}/{route}/metadata.ts`
2. Create shared implementation under `app/(i18n)/_shared/{route}/`
3. Create thin locale wrappers in `app/(i18n)/{en,ar,es}/{route}/page.tsx`
4. Add `setRequestLocale(locale)` at the top of each page component
5. Wire `generateMetadata` with `alternates.canonical` + `alternates.languages`
6. Add route to per-locale sitemaps

## Sitemaps

Separate sitemap per locale:

- `sitemap-en.xml` — English routes with hreflang alternates
- `sitemap-ar.xml` — Arabic routes with hreflang alternates
- `sitemap-profiles.xml` — `/@username` routes (locale-neutral)

Utilities in `src/lib/sitemap/sitemap-utils.ts`.

## Experts OS (iOS)

iOS i18n uses native SwiftUI localization. Locale is stored in `LocaleStore` and synced with the app preference. See [[Projects/Experts/Experts OS/docs/reference/Localization — Experts OS]].
