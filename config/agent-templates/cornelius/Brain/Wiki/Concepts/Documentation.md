# Documentation

Documentation in this vault is not passive reference material. It is the operating layer that keeps project decisions, runbooks, planning context, and agent instructions reusable across sessions.

## Documentation types that matter here

- project overviews and structure docs
- implementation guides
- release/version notes
- planning artifacts and roadmaps
- rules for agents and IDE tooling

## What good docs look like in this vault

- specific to a project, not generic filler
- updated when behavior or structure changes
- linked from the right entity/topic pages
- usable by both humans and coding agents

## Common gaps exposed by conversations

- implementation changed but the guide did not
- rules exist in prompts but not in canonical docs
- a topic page is only a pointer and lacks project decisions
- imported conversations contain the best explanation, but it was never distilled into a stable note

## Why this topic is central

Many maintenance tasks in the brain are really documentation hygiene:

- enriching thin topic pages
- wiring AGENTS/CLAUDE docs to real tool paths
- capturing architecture decisions before they disappear into chat history
