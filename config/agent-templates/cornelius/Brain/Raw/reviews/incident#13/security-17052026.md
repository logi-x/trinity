# Security Incident Remediation Workflow

## Discovery

Surfaced during the [[incident#10]] investigation pass (security-auditor + qa-tester
parallel run on 2026-05-17). qa-tester traced an HTML-injection sink in the three
share pages back to `getRequestBaseUrl`, which trusted attacker-controlled headers
without validation.

This incident was carved out of #10's scope and shipped as its own hotfix because:

1. It is a live vulnerability, not a structural migration.
2. The fix is small (4 files + 1 test file).
3. Bundling with #10's multi-PR phased CSP migration would delay the close.

## Security Review Incident #13

### Report

**Severity: High** (escalated from Medium during triage). Unauthenticated,
publicly reachable, broad phishing surface.

**Category:** Open redirect + near-XSS via host-header injection.

**Affected surface:**

- `/share/event/[id]`
- `/share/course/[id]`
- `/share/post/[id]`

All three share pages call `getRequestBaseUrl(headers)` from
`apps/experts-app/src/lib/share/share-utils.ts`. The old implementation:

```ts
const forwardedHost =
  headers.get("x-forwarded-host") ||
  headers.get("x-forwarded-server") ||
  headers.get("host");
if (!forwardedHost) return FALLBACK_BASE_URL;
const proto = headers.get("x-forwarded-proto") || "https";
return `${proto}://${forwardedHost}`;
```

Zero validation. Any HTTP client can set `x-forwarded-host: evil.com` and
have the resulting `canonicalUrl` flow into three sinks per share page:

| Sink                               | File                      | Old line | Attack                                                                                                                |
| ---------------------------------- | ------------------------- | -------- | --------------------------------------------------------------------------------------------------------------------- |
| `<script dangerouslySetInnerHTML>` | share/event/[id]/page.tsx | 114-118  | String interpolation into JS body — near-XSS (URL parser defangs most payloads but structural anti-pattern). DELETED. |
| `<meta http-equiv="refresh">`      | share/event/[id]/page.tsx | 113      | Refresh redirects to attacker host — open redirect. FIXED by hardening source.                                        |
| `<Link href>` CTA                  | share/event/[id]/page.tsx | 129, 135 | Clickable redirect to attacker host — open redirect. FIXED by hardening source.                                       |

(Same triplet in `course/[id]/page.tsx` and `post/[id]/page.tsx`.)

**Reachability gate:** depends on whether Traefik / Cloudflare strips
`x-forwarded-host` before it reaches the app. Conservative assumption:
live. Until verified, treat as exploitable in production.

### Root cause and full sink inventory

Audit performed on all 20 `getRequestBaseUrl` / `canonicalUrl` consumers:

| Consumer category                                                                            | Sites | Risk                                                                                            | Closed by this PR     |
| -------------------------------------------------------------------------------------------- | ----- | ----------------------------------------------------------------------------------------------- | --------------------- |
| Share-page `<script>` redirect                                                               | 3     | Near-XSS / open redirect                                                                        | Yes (deleted)         |
| Share-page `<meta refresh>`                                                                  | 3     | Open redirect                                                                                   | Yes (hardened source) |
| Share-page `<Link href>` CTA                                                                 | 6     | Open redirect on click                                                                          | Yes (hardened source) |
| Metadata `url` / `@id` / canonical in JSON-LD/OG (events/posts/users/courses metadata files) | ~19   | Metadata pollution (OG scraper poisoning) — React/Next escape values into attributes, so no XSS | Yes (hardened source) |

Single root cause → single fix → cascade closure.

### Remediation (shipped in PR #320, commit `e3bccd75`)

**`src/lib/share/share-utils.ts`:**

- New `STATIC_ALLOWED_HOSTS` array: `experts.com.sa`, `app.experts.com.sa`,
  `stg.experts.com.sa`, `app.stg.experts.com.sa`.
- `deriveAllowedHosts()` adds hosts parsed from `NEXT_PUBLIC_APP_URL` and
  `NEXT_PUBLIC_SITE_URL` (with try/catch on URL parse), plus
  `localhost` / `127.0.0.1` on common dev ports when `NODE_ENV !== "production"`.
- New exported `isAllowedRequestHost(host)` helper.
- `getRequestBaseUrl`:
  - Takes first hop only for comma-list `x-forwarded-*` (proxy chains).
  - Lowercases host before allowlist compare.
  - Validates `x-forwarded-proto` against `http`/`https`; defaults to `https`.
  - Returns `FALLBACK_BASE_URL` (canonical configured URL) when host is not in
    allowlist or no headers are present.

**Three share pages:** delete the redundant `<script>setTimeout(window.location.href=...)</script>`
block. The adjacent `<meta http-equiv="refresh">` already provides the redirect
with the same 1.5s delay budget. Net behavior preserved.

**`src/lib/share/__tests__/share-utils.test.ts`:** 13 unit tests covering:
allowed hosts (case-insensitive), attacker-controlled hosts, string-break attack
payloads, proxy chain first-hop-only semantics, header fallback chain
(`x-forwarded-host` → `x-forwarded-server` → `host`), bogus proto handling,
empty / nullish input.

### Open items

- **Verify reachability in production:** check Traefik config to confirm whether
  `x-forwarded-host` is preserved as-sent. If preserved, this hotfix closes a
  live vulnerability; if stripped, this hardens a defense-in-depth layer.
- **Smoke test on staging after PR #320 merges and deploys:**
  ```bash
  curl -sI -H 'x-forwarded-host: evil.com' https://stg.experts.com.sa/share/event/<valid-id> \
    | grep -i -E '(refresh|location)'
  ```
  Expected: the body's `<meta refresh>` URL points to the canonical staging
  host, NOT `evil.com`. The response should contain ZERO `<script>` redirect
  blocks (only the JSON-LD `<script type="application/ld+json">` remains).
- **Audit other apps:** the Experts platform has other apps (`experts-realtime`,
  `experts-prisma`, native iOS). None appear to read host headers for URL
  construction, but a sweep would confirm.

### Rules

Categorize:

- **XSS** (near; defanged in current behavior by URL parser)
- **business logic abuse** (open redirect → phishing)
- **data exposure** (metadata pollution downstream)

### CRITICAL RULES

NEVER allow this during security fixes:

- multiple writing agents editing same branch
- start implementation first

Always:

investigate → plan → implement → review
ALWAYS ask reviewers to verify reachability

### Review Process

1. Investigate the issue — security-auditor + qa-tester parallel (read-only)
2. Plan the fix — single-file hardening + 3 deletions + new test
3. Implement the fix — main session, ONE writer
4. Review the fix — self-review (user opted to skip 3-agent fan-out for a 4-file surgical change)

### Reviewers

- security-auditor (investigation phase, read-only)
- qa-tester (investigation phase, read-only)

### Review Criteria

- The fix is correct — yes (allowlist + 13 tests + clean check gate)
- The fix is secure — yes (closes all 3 cascading sinks at the root)
- The fix is effective — yes (single point of fix vs N JSX sites)
- The fix is maintainable — yes (env-derived allowlist, hardcoded baseline, dev hosts gated)
- The fix is tested — yes (13/13 unit tests covering attack matrix)
