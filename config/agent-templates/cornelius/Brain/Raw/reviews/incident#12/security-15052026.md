### Step 2 results (implemented)

**Surprising finding:** the original `uploadPublicAsset` already forced `Content-Disposition: attachment` for any non-`image/*`-non-`video/*` MIME via a prefix-match. The scope shift for Step 2 was therefore from "add behaviour" to "harden the existing behaviour against future drift" — particularly the silent SVG-XSS trap that the prefix-match would create if SVG were ever added upstream.

**Shipped:**

1. **Refactor:** `isImageOrVideo` (prefix-match) → `isInlineRenderableMime` (explicit 8-MIME allowlist). SVG, source code, and every other MIME now go through the attachment path by construction.
2. **RFC 5987 / RFC 6266 filename encoding** via new `buildContentDispositionHeader`. ASCII-safe names get the legacy `filename="..."` form; non-ASCII or special-character names additionally get `filename*=UTF-8''<encoded>` so Arabic / spaces / parentheses round-trip correctly.
3. **API change:** `uploadPublicAsset` now accepts optional `displayFilename` — the user-visible name shown in the browser's download dialog. The internal `/api/v1/internal/upload` route now passes the original `file.name` (length-capped, kept as-is for the download dialog) through.
4. **28-test suite** in `src/lib/storage/commands/__tests__/upload-public-asset.command.test.ts`:
   - 8 inline-renderable cases (no `Content-Disposition`).
   - 10 non-renderable cases (PDF, ZIP, JSON, text, audio, Office) → `attachment`.
   - SVG explicitly tested as attachment-served (XSS guard).
   - 6 future source-code MIMEs (`application/javascript`, `text/x-python`, etc.) → confirmed attachment-served _before_ Step 3 adds them to the upload allowlist.
   - RFC 5987 cases: Arabic, spaces, RFC-reserved chars (`'`, `(`, `)`, `*`).
5. **Vault hardening:** `~/brain/Projects/Experts/Experts App/docs/security/cdn-hardening.md` Section 4 extended with the inline-renderable allowlist + Step 3 forward-compatibility note.

**Hard rule documented in vault:** any new MIME added to the upload validator that is NOT in the inline-renderable allowlist MUST be served as `attachment` by `uploadPublicAsset` — protected by the test suite.

**Step 3 simplification:** because of Step 2's allowlist refactor, Step 3 (adding source-code MIMEs) requires NO `uploadPublicAsset` changes — only `ALLOWED_FILE_TYPES` and `MIME_TO_EXT` extensions in the upload route.

**Outstanding (small, ops-side):** CDN round-trip verification curl per `cdn-hardening.md` Section 4 — run after Step 2 ships and ideally against a newly-uploaded `.pdf` with a non-ASCII filename to confirm Cloudflare passes both `filename=` and `filename*=` parameters verbatim. ## Security Review Incident #12 (Enhancement)

### Report

**Category:** enhancement with security implications. Not a remediation of a current vulnerability — a feature request that requires deliberate security design before it ships.

Surfaced during the PR #317 smoke test for incident #9 follow-up. The user observed: _"in some cases a lesson might contain a .js as an asset of that lesson"_ — e.g., a "Learn JavaScript" course needs to attach `.js` source files for students to download.

Today's state: `/api/v1/internal/upload`'s `ALLOWED_FILE_TYPES` does NOT include `application/javascript`, `text/javascript`, or any source-code MIME. So `.js` files cannot be uploaded through the platform at all. This is safe — but blocks a legitimate product capability.

### The threat model that needs solving

If we simply add `application/javascript` to ALLOWED_FILE_TYPES + `MIME_TO_EXT["application/javascript"] = "js"`, the platform begins:

1. Storing attacker-uploaded `.js` files on `cdn.experts.com.sa` at predictable keys
2. CDN serves them with `content-type: text/javascript`
3. A direct navigation to the URL executes the JS in the `cdn.experts.com.sa` origin (not the app origin)

Three independent threat layers, current state per the 2026-05-14 production CSP audit:

| Threat                                                                                 | Currently mitigated by                                                                        | Risk if mitigations regress                     |
| -------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| XSS on app origin via `<script src="https://cdn.experts.com.sa/...evil.js">`           | App CSP `script-src` does **not** include `cdn.experts.com.sa`                                | **Critical XSS** if CSP ever loosens            |
| Cookie theft on `experts.com.sa` parent domain when user visits the `.js` URL directly | Cookies presumably scoped to `app.experts.com.sa`, not `.experts.com.sa` (needs verification) | **Session hijack** if cookies are parent-scoped |
| Phishing / platform-credibility                                                        | None — anyone can claim "see attached `https://cdn.experts.com.sa/.../legit-looking.js`"      | **Medium** — social engineering surface         |

### Recommended remediation path (multi-step)

This is a 3-step migration:

#### Step 1: Audit and document parent-domain cookie scope

Run:

```bash
curl -sI 'https://app.experts.com.sa/api/v1/auth/session' | grep -i 'set-cookie'
curl -sI 'https://experts.com.sa/' | grep -i 'set-cookie'
```

Confirm every `Set-Cookie` line uses `Domain=app.experts.com.sa` (not `Domain=.experts.com.sa`). If any cookie is parent-scoped, fix that **before** any of steps 2-3 — otherwise a `.js` lesson asset on the CDN can read the session cookie when the user navigates to it directly.

#### Step 2: Force `Content-Disposition: attachment` for source-code resources

Modify `uploadPublicAsset` (or `/api/v1/internal/upload` directly) to set `Content-Disposition: attachment; filename="<safe-filename>"` on the R2 PUT when the upload's `domain` is `lesson-resources` or `module-assets` AND the MIME indicates a non-renderable type (source code, archives, executables). The browser will then download the file instead of executing it on the CDN origin.

Verify the header survives the Cloudflare round-trip — incident #7 already includes a runbook checklist item for this.

#### Step 3: Add source-code MIMEs to the allowlist for specific domains only

Extend `ALLOWED_FILE_TYPES` selectively. Two options:

- **Option A (single allowlist, MIME-keyed):** add `application/javascript` (and friends: `text/x-python`, `text/x-typescript`, `application/x-tar`, etc.) to the existing global `ALLOWED_FILE_TYPES`. Simpler.
- **Option B (per-domain allowlists):** introduce per-domain allow tables, so `lesson-resources` and `module-assets` can accept source code while `community` and `draft-courses` cannot. Tighter scope; recommended.

Map each new MIME to its real extension in `MIME_TO_EXT`. With the incident #9 fix in place, the key now correctly ends in `.js` (not the attacker filename), so the structural invariant ("extension matches validated MIME") is preserved.

#### Step 4: Optional — separate hostname

Long-term, consider serving lesson resources from a **different subdomain** (e.g., `files.experts.com.sa`) that is intentionally never in any `script-src` allowlist. This adds another layer of isolation between attacker-uploaded source code and any future CSP regression.

### Why this is its own incident, not a quick add

- Cookie-scope verification (Step 1) is a hard prerequisite — getting Step 3 without Step 1 is a session-hijack opportunity for any attacker who can upload a `.js` lesson resource.
- Content-Disposition handling (Step 2) cuts across `uploadPublicAsset` and the CDN config (incident #7's runbook).
- Per-domain allowlists (Step 3 Option B) is a refactor of the upload route's validation shape, not a one-line change.
- Each step is independently shippable AND verifiable.

### Affected files (initial estimate)

- `apps/experts-app/src/lib/storage/commands/upload-public-asset.command.ts` — accept optional `contentDisposition` parameter.
- `apps/experts-app/app/api/v1/internal/upload/route.ts` — when domain ∈ {`lesson-resources`, `module-assets`} and MIME is non-renderable, pass `contentDisposition: 'attachment; filename="..."'`.
- `apps/experts-app/app/api/v1/internal/upload/route.ts` — extend `ALLOWED_FILE_TYPES` and `MIME_TO_EXT`.
- `~/brain/Projects/Experts/Experts App/docs/security/cdn-hardening.md` — extend the Content-Disposition round-trip check to specifically cover lesson resources.
- (Optional) DNS / Cloudflare config for `files.experts.com.sa`.
