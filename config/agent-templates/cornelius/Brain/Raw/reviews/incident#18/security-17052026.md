You are the main security incident orchestrator for the Experts project.

I will provide a list of security incidents. Run the full process end-to-end in this order:

1. Investigate
   Spawn/read from:
   - security-auditor
   - dependency-auditor if dependencies/CVEs are involved
   - architecture-reviewer
   - qa-tester

2. Triage
   For each incident classify:
   - severity
   - exploitability
   - reachability
   - affected files/systems
   - whether it is hotfix, normal fix, or false positive

3. Plan
   Ask the relevant specialist to propose the safest remediation:
   - frontend-developer for frontend/API route fixes
   - db-engineer for Prisma/Postgres/migration issues
   - payments-specialist for ZATCA/payments/ledger issues
   - realtime-engineer for WebSocket/presence issues
   - devops-engineer/platform-operator for infra/config issues

4. Implement
   Use exactly ONE writing agent.
   Do not allow multiple agents to edit files.
   Work only in the current directory.
   Do not create worktrees.
   Do not switch branches.
   Do not push until review passes.

5. Review
   Spawn read-only review from:
   - security-auditor
   - code-reviewer
   - qa-tester
   - architecture-reviewer if architectural changes were made
   - release-manager if migrations/config/deploy changes were made

6. Fix review findings
   Apply only necessary changes.
   Avoid broad refactors or unrelated cleanup.

7. Validate
   Run relevant tests/typechecks/lints for the changed scope.
   If a command fails, explain whether it is caused by this change or existing project state.

8. Commit and push
   Only after review and validation.
   Make clean, logical commits.
   Push the current branch only.
   Never force-push.

Before doing anything, print:

- git status -sb

## Security Review Incident #8

### Report

Scheduled application security review: 3 validated medium+ findings

1. High: password-reset token URL poisoning via attacker-controlled origin
• Location: apps/experts-app/src/lib/utils.ts, apps/experts-app/app/api/auth/forgot-password/route.ts, also app/api/v1/auth/native/forgot-password/route.ts
• Attacker/input: unauthenticated caller controls Origin, Referer, Host, or x-forwarded-proto headers plus the victim email in the forgot-password request.
• Reach path: POST /api/auth/forgot-password creates a PASSWORD_RESET token, calls getBaseUrl(request), then emails resetUrl built from those headers.
• Impact: victim receives a legitimate reset email whose link can point to an attacker domain with the reset token in the query; if clicked, the attacker can capture the token and reset the victim’s password via POST /api/auth/reset-password.
• Remediation: build secret-bearing URLs only from a configured canonical origin / allowlisted tenant origin; do not trust request Origin/Referer/Host for reset, verify, checkout, or enrollment callback URLs.

2. Medium: account enumeration on web forgot-password endpoint
• Location: apps/experts-app/app/api/auth/forgot-password/route.ts
• Attacker/input: unauthenticated caller controls JSON email.
• Reach path: POST /api/auth/forgot-password returns distinct 404, 409, 429, or 200 responses for unknown emails, Google-only accounts, non-password accounts, cooldown state, and valid password accounts.
• Impact: attacker can map valid users and auth-provider type for targeted phishing or credential stuffing.
• Remediation: use the native route’s safer pattern: always return the same generic success response for syntactically valid email input; keep cooldown/provider decisions internal and add IP/email-hash throttling.

3. Medium: unauthenticated detailed console health disclosure
• Location: apps/experts-app/app/api/v1/console/health/route.ts
• Attacker/input: any unauthenticated client sends GET /api/v1/console/health.
• Reach path: handler runs DB and Redis checks and returns environment, uptime, version/image/sha/release identifiers, integration configuration status, subsystem map, and dependency error details without auth() or requireAdmin().
• Impact: exposes production fingerprinting and dependency readiness details useful for targeted follow-on attacks and timing.
• Remediation: gate with requireAdmin() or a dedicated service health token; keep only a minimal public liveness endpoint if required.

Other reviewed areas included API auth/RBAC samples, webhooks, raw SQL, shell/file sinks, uploads, SSRF-like fetches, markdown/HTML rendering, and realtime auth. Conditional forwarded-host share redirects and unwired proxy.ts middleware were not reported as standalone validated medium+ issues. No PR opened and no code changes made.
