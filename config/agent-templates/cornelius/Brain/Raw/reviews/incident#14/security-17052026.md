## Security Review Incident #14 (Developer Experience)

### Discovery

Surfaced in the developer console after incident #10 Phase 4B (PR #324) landed:

```
installHook.js:1 eval() is not supported in this environment.
If this page was served with a `Content-Security-Policy` header,
make sure that `unsafe-eval` is included.
React requires eval() in development mode for various debugging
features like reconstructing callstacks from a different environment.
React will never use eval() in production mode.
```

The error originates from React DevTools / React's dev runtime — not from
production code paths. React's own message confirms this: `eval()` is dev-only.

### Category

**Developer experience** (not a production security risk). The CSP is doing
exactly what it should in production: blocking `eval()`. The problem is that
the same enforcing CSP also runs in `next dev`, where React's dev features
need `eval()` for source-map reconstruction, error-overlay callstacks, hot
reload, and React DevTools hook installation.

### Current state (post #324)

`apps/experts-app/proxy.ts` builds ONE `CSP_DIRECTIVES` constant and ships it
verbatim across all environments (dev, staging, prod):

```ts
"script-src": [
  "'self'",
  "https://elu.dev",
  "https://static.cloudflareinsights.com",
  "https://www.googletagmanager.com",
  "https://us-assets.i.posthog.com",
],
// nonce-${randomUUID} is injected at request time, but no 'unsafe-eval'
```

`grep -n "NODE_ENV\|development\|production" apps/experts-app/proxy.ts` returns
only the directive-name strings — no environment branching exists.

### What's broken

In `pnpm dev`:

1. React DevTools' `installHook.js` cannot register on `window` → DevTools
   panel says "This page doesn't appear to be using React."
2. The React error overlay loses cross-frame source-mapping → unhelpful stack
   traces in dev.
3. Fast Refresh / HMR can fail silently in some build configurations.
4. Anything that calls `new Function(...)` (some dev-only validators,
   Prisma's debug paths, certain MUI/HeroUI dev helpers) trips a CSP violation.

### What's NOT broken

- Production runtime — React 19 production build does not use `eval()` or
  `new Function()`. Removing `'unsafe-eval'` from prod CSP is the **correct**
  hardening that Phase 4B achieved.
- Staging — same as prod; staging should remain locked down.
- The nonce-based script-src model — nonces still flow correctly; this is
  orthogonal to the nonce architecture.

### Threat model — why dev-only `'unsafe-eval'` is acceptable

`'unsafe-eval'` allows `eval(string)`, `new Function(string)`, and similar
string-to-code conversions. The risk model:

| Environment                       | Attack surface for `'unsafe-eval'` abuse                                                             | Verdict              |
| --------------------------------- | ---------------------------------------------------------------------------------------------------- | -------------------- |
| Prod                              | Attacker-controlled string + XSS sink → arbitrary code execution. **Must NOT have `'unsafe-eval'`.** | Keep locked          |
| Staging                           | Same risk shape as prod (real users may visit it, real third-party scripts run).                     | Keep locked          |
| Dev (localhost, single developer) | No untrusted users on `localhost:3025`; React itself is the only `eval()` caller.                    | Acceptable to enable |

Conclusion: gate `'unsafe-eval'` strictly on `process.env.NODE_ENV !==
"production"` AND not on `staging` (the staging app is built with
`NODE_ENV=production` regardless of branch name, so the prod check covers it).

### Recommended remediation

#### Step 1: Conditional `'unsafe-eval'` in dev only

`apps/experts-app/proxy.ts` — make the `script-src` directive environment-aware:

```ts
function buildScriptSrc(nonce: string): string {
  const sources = [
    "'self'",
    `'nonce-${nonce}'`,
    "https://elu.dev",
    "https://static.cloudflareinsights.com",
    "https://www.googletagmanager.com",
    "https://us-assets.i.posthog.com",
  ];
  if (process.env.NODE_ENV !== "production") {
    sources.push("'unsafe-eval'");
  }
  return `script-src ${sources.join(" ")}`;
}
```

Equivalent shape: keep `CSP_DIRECTIVES` purely declarative and add the
`'unsafe-eval'` source via a small `applyDevModeRelaxations(directives)`
function. Either is fine — the inline branch is easier to audit at a glance.

#### Step 2: Empirical truth-test

After the dev server restarts:

```bash
curl -sI http://localhost:3025/en \
  | grep -i 'content-security-policy:' \
  | grep -oE "script-src [^;]+"
```

Expected in dev:

```
script-src 'self' 'nonce-<base64>' 'unsafe-eval' https://elu.dev ...
```

Expected in prod (verified post-deploy against `app.experts.com.sa`):

```
script-src 'self' 'nonce-<base64>' https://elu.dev ...
# (NO 'unsafe-eval')
```

#### Step 3: Regression test

`apps/experts-app/__tests__/csp-environment.test.ts` (new):

- Set `process.env.NODE_ENV = "production"`, call `buildAppCsp(nonce)`,
  assert the returned string does NOT contain `'unsafe-eval'`.
- Set `process.env.NODE_ENV = "development"`, call again, assert it DOES
  contain `'unsafe-eval'`.
- Set `process.env.NODE_ENV = "test"`, assert it DOES contain
  `'unsafe-eval'` (vitest runs under `NODE_ENV=test`; matches dev posture).

This test locks the prod-hardening invariant — any future refactor that
accidentally enables `'unsafe-eval'` in production fails CI.
