# Security Incident Remediation Workflow

## Discovery

Surfaced during smoke-testing of incident #12 Step 3 (per-domain source-code MIME allowlist). User attempted to upload images through three creator-curriculum panels and reported "nothing happens, no sign of anything, no error, response, request, toast." Triage uncovered four distinct bugs in the creator-curriculum upload flow, all silent or returning misleading `403 Forbidden`:

1. Lesson-resources panel silently swallowed images at the picker / dropzone layer.
2. Course-assets and module-assets dropzones silently swallowed dropped images (button-click path worked).
3. Quiz-question image uploads returned `403 Forbidden` because the UI sent client-generated UUIDs for unsaved questions.
4. Lesson-resource creation failed with `Unique constraint failed on (lesson_id, role, position)` because position was derived from the wrong table.

## Category

**Business logic + authorization** (not a remediation of a current vulnerability — the auth checks themselves are correct; the bugs are in how the UI invokes them, and in one case how the handler derives a uniqueness key).

## Bug 1 — Lesson-resource picker excluded images and forced `type=file`

**Where:** `apps/experts-app/app/(i18n)/_shared/creator/courses/[id]/curriculum/sections/lesson-resource-panel.tsx`

The hidden `<input type="file" accept=".pdf,.zip,.doc,…">` had no `image/*` entry — Chrome silently filtered images out of the picker, so no `onChange` ever fired. Even if a user dragged an image in, `formData.append("type", "file")` was hardcoded, so the server would have returned 400 (mismatched MIME class) had the file ever reached upload.

**Fix:** `accept="image/*,video/*,audio/*,.pdf,.zip,…"`; derive `uploadType` from `file.type` (`image` / `video` / `file`) like the sibling panels already do.

## Bug 2 — All three dropzones rejected images via `remainingImageSlots={0}`

**Where:** three files in `apps/experts-app/app/(i18n)/_shared/creator/courses/[id]/curriculum/sections/`:

- `lesson-resource-panel.tsx`
- `course-assets-panel.tsx`
- `module-assets-panel.tsx`

`AssetDropzone` splits accepted files into `images` (matching `image/*`) and `files` (everything else). With `remainingImageSlots={0}`, the `images` array was sliced to length 0; the `files` array filtered out the image too (the dropzone's filter is `f => !f.type.startsWith("image/")`). Net result: an image drop produced no callback or an empty callback that all three panels ignored (`(_, files) => handleFileUpload(files[0] ?? null)` — the `images` array was discarded).

**Fix per panel:** `remainingImageSlots={1}` + `onFilesAccepted={(images, files) => handleFileUpload(files[0] ?? images[0] ?? null)}`.

## Bug 3 — Quiz-question image upload returned 403 (unsaved questions have no DB id)

**Where:** `apps/experts-app/app/(i18n)/_shared/creator/courses/[id]/curriculum/sections/quiz-dialog.tsx`

The dialog's `addQuestion` function generates a client-side UUID for each new question (line 217). Questions are local form state until the user clicks Save. Image uploads were posting `domain=quiz-questions, entityId=<client-uuid>` to `/api/v1/internal/upload`. The route authorizes via `prisma.quizQuestion.findFirst({id: entityId, quiz: {course: {instructors}}})` — that row doesn't exist yet, so every upload returned 403 (NOT 400, NOT 404, just "Not authorized").

**Symptom that misled triage:** user reported it as a security/authz regression, but the auth check is correct — the entityId simply never existed in DB.

**Fix:**

- `apps/experts-app/app/api/v1/internal/upload/route.ts` — new `quiz` case in `authorizeDomainAccess` querying `prisma.quiz.findFirst({id: entityId, course: {instructors}})`. Added `"quiz"` to `ALLOWED_DOMAINS`.
- `quiz-dialog.tsx` — upload now posts `domain=quiz, entityId=editingQuiz.id`. Asset is stored at `uploads/quiz/<quizId>/<uuid>.<ext>` and the resulting URL is appended to the question's `imageUrls` array on save (existing flow).
- `QuestionAssetUploader` gets `disabled={!editingQuiz?.id}` so the picker can't fire before the quiz exists. Belt-and-braces guard in the upload handler logs and early-returns if a stale render still tries.
- 2 new tests in `apps/experts-app/app/api/v1/internal/upload/__tests__/route.test.ts` covering `domain=quiz` 200/403 paths.
- Existing `quiz-questions` domain is **not removed** — it remains valid for any future flow that uploads against an already-saved question.

## Bug 4 — Lesson-resource creation collided on lesson_assets unique constraint

**Where:** `apps/experts-app/src/lib/courses/curriculum/lessons/handlers/resources/course-lesson-resource-create.handler.ts`

The handler derived `position` from `lesson_resources.max(position) + 1`. But the unique constraint `@@unique([lessonId, role, position])` lives on **`lesson_assets`**, not `lesson_resources`. Whenever those two tables drift out of sync — orphan `lesson_assets` rows with `role="resource"` from past partial failures, manual cleanup, or migrations — the parallel `lessonAsset.create` inside the `$transaction` collides:

```
Unique constraint failed on the fields: (`lesson_id`, `role`, `position`)
```

Surfaced on a real lesson the user had previously cleaned up; the upload to R2 succeeded, then the `createResource` API call failed with 500, leaving an orphan CDN object.

**Fix:** take `Math.max(lessonResource.max(position), lessonAsset.max(position) where role=resource) + 1`. Both queries run in parallel; both inserts use the same position so the two tables stay in lockstep going forward. New regression test: when `lessonAsset` has higher max than `lessonResource`, the new row uses the `lessonAsset`-derived position.

## Threat model — not a security regression

None of these bugs lower the security posture:

- Auth checks on the upload route still enforce course ownership.
- The new `quiz` domain authorizes against quiz → course → instructor, no weaker than the per-question check it replaces (and identical in effective scope, since a question can only be uploaded by someone who owns the parent course).
- The dropzone change exposes a UI affordance that was always intended to be available; it just wasn't wired up.
- The handler position fix prevents an integrity error, not a security one.

## Operational lessons

1. **403 doesn't always mean "auth regression."** If a route's authz is a Prisma `findFirst` against the entity ID, 403 can also mean "entity doesn't exist" — especially when the UI uses optimistic client-side UUIDs.
2. **Silent dropzone rejection is a UX trap.** Components like `AssetDropzone` that filter by MIME and slice to slot counts will swallow inputs with zero visible feedback. Either log + toast on rejected drops, or gate the UI at the affordance level (don't show a slot you won't accept).
3. **Unique-constraint derivation across denormalized tables is fragile.** When two tables MUST stay in lockstep (here: `lesson_resources` and `lesson_assets` for resources), derive shared keys from the table that owns the constraint — or from the max of both.
4. **Hardcoded `accept=` attributes drift from the route's allowlist.** Either generate `accept=` from a shared MIME constant, or accept everything client-side and rely on the route's validation. Today's app does neither — each panel maintains its own `accept=` string by hand.
