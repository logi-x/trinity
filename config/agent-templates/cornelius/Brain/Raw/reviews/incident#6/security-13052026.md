## Security Review Incident #6

### Report

Carried forward from incident #2 (PR #308). Defense-in-depth gap surfaced by qa-tester.

Low: declared `file.size` trusted over actual buffer length in upload route

- **Location:** `apps/experts-app/app/api/v1/internal/upload/route.ts` — the `if (file.size > MAX_FILE_SIZE)` check at the request validation step, and the `prisma.file.create({ data: { size: file.size, ... } })` insert.
- **Attacker:** any authenticated user (post-fix, must also be an owner of the target entity).
- **Path:** the 50 MB cap is evaluated against `file.size`, which is the size declared by the multipart parser based on the client-supplied stream metadata. The actual `await file.arrayBuffer()` is materialized into memory before any further check. If the platform layer (Next.js runtime / undici) ever fails to enforce its own body-size limit, an oversize request could be materialized in memory before the route notices.
- **Likely status:** Next.js App Router has a default body-size limit that typically rejects oversize multipart requests before the handler is even invoked, so this is mostly defensive. Worth confirming the actual configured limit.
- **Highest-leverage remediation:**
  - Re-check size against `buffer.byteLength` immediately after `arrayBuffer()` and return 413 if it exceeds `MAX_FILE_SIZE`. Cheap, two-line change.
  - Use `buffer.byteLength` (not `file.size`) when writing the `File` row, so the DB record reflects truth.
  - Audit the Next.js / route-segment `bodySizeLimit` config (e.g., `export const runtime = ...`, `export const maxDuration = ...`, or middleware-level size enforcement) and document the effective limit somewhere agents can find it. If it's higher than 50 MB at the platform layer, lower it.
- **Affected paths:** just the one route; no callers need to change.

This is a small, surgical fix. Single writing agent, single file edit, plus optional config verification.
