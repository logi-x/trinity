## Security Review Incident #7

### Report

Open question from incident #2 (PR #308). This is an infra/config audit task, not an app-code fix — most likely handled by **devops-engineer** rather than frontend-developer.

Medium: confirm CDN and CSP hardening on `cdn.experts.com.sa`

The incident #2 fix relies on `Content-Disposition: attachment` to neutralize the residual risk of attacker-influenced MIME on non-media uploads. That mitigation is only as strong as the CDN configuration that serves the asset. Two specific items to verify and, if missing, fix:

1. **`X-Content-Type-Options: nosniff` on every CDN response.**
   - **Why:** without this header, some browsers will sniff the body and override the declared `Content-Type`. A file uploaded as `text/plain` whose body is HTML could then be rendered as HTML by a sniffing browser.
   - **Action:** verify Cloudflare R2 / Cloudflare proxy / whatever fronts `cdn.experts.com.sa` sets `X-Content-Type-Options: nosniff` on **all** responses from the bucket. If missing, add via a Cloudflare Transform Rule or Page Rule on `cdn.experts.com.sa/*`.

2. **App CSP does NOT allowlist `cdn.experts.com.sa` as a `script-src`.**
   - **Why:** if the app's CSP allows `script-src https://cdn.experts.com.sa`, any attacker-uploaded `.js` file on the CDN becomes executable on the app origin. (Incident #2 removed `text/javascript` from the MIME allowlist, but that's app-layer defense — CSP is the platform-layer defense.)
   - **Action:** read the response CSP header from a few production pages (`app/...`), confirm `cdn.experts.com.sa` appears only in `img-src` / `media-src` / `connect-src` (as appropriate), never in `script-src` or `default-src`. Lock down if drift is found.

3. **Bucket listing disabled on the R2 public bucket.**
   - **Why:** flagged by qa-tester. The `draft-courses` storage path is `uploads/draft-courses/<userId>/...`. If bucket listing is enabled (publicly), anyone who knows or guesses a userId can enumerate that user's draft uploads.
   - **Action:** verify Cloudflare R2 public bucket has listing disabled (`ListBucket` blocked at the bucket policy level for anonymous principals).

4. **`Content-Disposition: attachment` round-trips through the CDN intact.**
   - **Why:** if a Cloudflare Worker, Transform Rule, or cache layer ever strips response headers, the app-layer mitigation collapses.
   - **Action:** curl a non-media object from production and confirm `Content-Disposition: attachment; filename="..."` is present in the response. Add a synthetic monitor (or document a manual check in the runbook) so a future config change can't silently strip it.

This is a discovery + (possibly) configuration task. May result in zero code changes and only Cloudflare dashboard / Terraform / IaC updates — which still belong on a remediation branch if the IaC is in this repo, otherwise tracked in the infra repo and linked here.
