## Security Review Incident #5

### Report

Carried forward from incident #2 (PR #308). qa-tester surfaced this during review of the upload remediation; explicitly scoped out because it requires a transactional refactor rather than a surgical security fix.

Medium: orphaned R2 objects on partial-failure path in upload route

- **Location:** `apps/experts-app/app/api/v1/internal/upload/route.ts` (the sequence around `uploadPublicAsset` → `prisma.file.create` → `prisma.attachment.create`).
- **Sequence today (post-incident #2 fix):**
  1. Validate + authorize.
  2. Run `imageSize(buffer)` for images (now wrapped in try/catch so this step alone no longer leaks objects).
  3. `uploadPublicAsset(...)` — R2 PUT.
  4. `prisma.file.create(...)`.
  5. `prisma.attachment.create(...)`.
  6. Return success JSON.
- **Failure modes that orphan an R2 object:**
  - `prisma.file.create` throws (constraint violation, connection drop, schema drift). The R2 object exists; no `File` or `Attachment` row tracks it.
  - `prisma.attachment.create` throws after `file` is created. The `File` row exists but no `Attachment` references it.
- **Impact:** untracked objects accumulate in R2 — slow, persistent cost leak; impossible to GC because no DB record points at the orphaned key. Also a data-integrity drift between R2 and Postgres.
- **Highest-leverage remediation options:**
  - **Option A (recommended):** create `File` + `Attachment` rows in Postgres FIRST (status: `pending`), THEN PUT to R2, THEN flip status to `ready` in a final update. If the PUT fails, the row stays `pending` and a janitor job can delete pending rows older than N minutes from both R2 and DB. Idempotent retries become possible.
  - **Option B:** wrap PUT + DB writes in a try/catch with a compensating `DeleteObjectCommand` against R2 in the catch. Simpler diff, but compensating delete itself can fail and is not idempotent under retries.
  - **Option C:** queue the upload metadata into an outbox table inside a Prisma transaction with the file/attachment rows, then a worker performs the PUT against R2. Decouples request from R2 latency but is a bigger refactor.
- **Affected callers:** all 9 upload sites identified in incident #2's architecture review (course-details, course-assets, module-assets, lesson-resources, lesson-content, quiz-questions, exam-questions, community new-post, community edit-post).

This is a correctness/operational issue more than a security issue per se, but it ties into the storage hardening workstream from incident #2 and should be picked up by db-engineer + devops-engineer.
