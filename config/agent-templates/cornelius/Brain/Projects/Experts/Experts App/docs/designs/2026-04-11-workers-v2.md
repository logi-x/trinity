4️⃣ Why Prisma ended up in the bundle (the real reason)

This line from earlier explains everything:

const invoice = await prisma.invoice.findUnique({
include: invoicePdfInclude,
});

That means:

Your PDF adapter is still:

querying the database

depending on Prisma types

pulling generated/prisma transitively

So tsup must bundle Prisma runtime.

This is not tsup’s fault.

5️⃣ The correct rule (non-negotiable)

Workers must never import Prisma — directly or indirectly

That means:

❌ Worker → Adapter → Repository → Prisma
✅ API / Service → Repository → Prisma
✅ Worker → Adapter → Pure data

6️⃣ The fix (small but powerful)
Move this OUT of the worker path:
// ❌ not in worker execution path
prisma.invoice.findUnique(...)

Replace with:
// ✅ done before enqueueing job
const invoice = await getInvoiceForPdf(prisma, invoiceId);

enqueuePdfJob({
invoice, // plain object
});

Then in the worker:
processPdfJob(job) {
generateInvoicePdf(job.invoice);
}

Once you do this:

Prisma disappears from the bundle

@prisma/\* imports vanish

Bundle size drops significantly

Worker becomes portable (can run anywhere)

7️⃣ About bundle size (realistic expectations)

Even after cleanup:

2–4k LOC is normal for:

React PDF

AWS SDK

QR generation

What matters is:

no DB client

no framework

no Next.js runtime

Size ≠ problem
Hidden coupling = problem
